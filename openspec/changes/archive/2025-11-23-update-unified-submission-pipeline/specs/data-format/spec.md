## MODIFIED Requirements
### Requirement: 标准化的 MarkObject 结构
所有页面提交的数据 MUST 使用标准 `MarkObject`，字段与编码规则更新以统一 Flow/子模块的页码与前缀。
#### Scenario: MarkObject 字段与格式
- THEN `MarkObject` SHALL 包含：`pageNumber`、`pageDesc`、`operationList`、`answerList`、`beginTime`、`endTime`、`imgList`（可为空数组）；
- AND `pageNumber` SHALL be generated by `encodeCompositePageNum(stepIndex, subPageNum)` with format `<stepIndex>.<subPageNum>`（e.g., `0.3`）；`M*` 前缀与冒号分隔（如 `M2:11`）不再允许；
- AND `pageDesc` SHALL 使用 `[${flowId}/${submoduleId}/${stepIndex}] 页面描述` 前缀；缺失 Flow 时仅保留页面描述；
- AND `operationList/answerList` 中的 `targetElement` SHALL start with `P${pageNumber}_<业务ID>`，用相同的 `pageNumber` 前缀标识页面。

### Requirement: Operation 结构与事件类型
所有操作事件 MUST 符合统一结构与事件约束，提交层负责补全时间、code、上下文。
#### Scenario: Operation 字段
- THEN `Operation` SHALL 包含：`code: number`（从 1 自增，由提交层生成）、`targetElement: string`、`eventType: string`、`value: string | object`（可序列化）、`time: string`（`YYYY-MM-DD HH:mm:ss`），可选 `pageId: string`；
- AND 提交层 SHALL 统一生成时间戳（缺失时兜底），并保证 `Operation.code` 连续自增；子模块 MUST NOT 手写 code 或格式化时间。
#### Scenario: 标准事件类型与约束
- THEN `eventType` SHALL 选自更新后的集合：`page_enter`、`page_exit`、`click`、`change`、`input`、`input_blur`、`input_focus`、`input_change`、`input_delete`、`select_change`、`radio_select`、`checkbox_check`、`checkbox_uncheck`、`modal_open`、`modal_close`、`view_material`、`timer_start`、`timer_stop`、`timer_complete`、`simulation_timing_started`、`simulation_run_result`、`simulation_operation`、`questionnaire_answer`、`flow_context`、`auto_submit`、`next_click`、`page_submit_success`、`page_submit_failed`、`click_blocked`；扩展事件 MUST 经 Data Format Spec 评审同步更新枚举；
- AND `flow_context` SHALL 由提交层自动注入一次，`value` 包含 `flowId/stepIndex/submoduleId/moduleName`（可含 `pageId`）；缺失 Flow 信息时可跳过但需记录日志；
- AND `click_blocked` events SHALL include a `missing` string[] within `value` to 标识未满足的必填或校验项。
#### Scenario: 输入与选择事件粒度
- THEN 输入类事件 SHALL 捕获 `input_focus`（进入输入框）、`input_change`（内容变更，value 使用快照或 `{ prev, next }`）、`input_delete`（显式记录删除，value 推荐 `{ prev, next }` 或 `{ action: 'delete', prevLength, nextLength }`）、`input_blur`（离开输入框），以还原输入/删除链路；
- AND 选择类事件（单选/多选/下拉）SHALL 使用 `select_change`/`radio_select`/`checkbox_*`，value 记录当前选中项列表或具体选项标识；
- AND “下一页”操作 SHALL 记录 `next_click`（或等效 click 且 value 标识下一步），与 `page_exit`/提交事件对齐。
#### Scenario: 输入类 value 结构示例
- THEN `input_change` MAY use `{ prev: string, next: string }` 或 `{ prevLength: number, nextLength: number, delta?: string }` 以反映增删内容；
- AND `input_delete` SHOULD use `{ action: 'delete', prev: string, next: string }` 或 `{ action: 'delete', prevLength: number, nextLength: number }`；若与 `input_change` 合并，需至少提供前后快照或长度变化。

### Requirement: Answer 结构
问卷或页面答案 MUST 遵循固定结构，并仅记录有效答案。
#### Scenario: Answer 字段与过滤
- THEN `Answer` SHALL 包含：`code: number`（从 1 自增，由提交层生成）、`targetElement: string`（前缀 `P${pageNumber}_...`）、`value: string`；
- AND `answerList` SHALL 仅包含非空答案；空字符串/null/undefined SHALL NOT 入列；复杂数据（如实验历史、截图元数据） MAY 作为单条 Answer，`value` 为 JSON 字符串；
- AND 超时场景由提交层补齐“超时未回答”后再入列。

### Requirement: 提交前校验（前端）
提交链 MUST 在前端完成结构化校验以阻断不合规数据。
#### Scenario: Schema 校验与阻断
- WHEN 构建 `MarkObject` 准备提交
- THEN 提交层 SHALL 使用 TS 类型 + Zod/JSON Schema 校验必需字段、事件枚举、code 连续性、`pageNumber/pageDesc/targetElement` 新前缀规则与时间格式；不合规则 MUST 阻断并返回可读错误。

## ADDED Requirements
### Requirement: 页面行为覆盖度
每个页面的操作列表 MUST 覆盖最小必要事件集合，确保可还原用户行为轨迹。
#### Scenario: 最小事件集合
- THEN 每页 SHALL 至少包含 `page_enter`、`page_exit`、`next_click`（或等效 click 标记下一步），并在阻断时追加 `click_blocked`（含缺失项列表）；
- AND 输入类控件 SHALL 记录 focus/blur/change/delete 序列，选择/切换类控件 SHALL 记录每次变更；实验/计时类页面 SHALL 记录参数调整、计时开始/结束、运行结果。
