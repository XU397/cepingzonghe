## 1. 基本信息（Basic Info）[必填]

> 说明：本节用于快速识别子模块的业务属性与技术身份。**为便于直接替换现存子模块，名称与 ID 与现有 `g4-experiment` 保持一致。**

- **子模块名称（中文）**：`4年级火车购票-交互`
- **子模块标识 `submoduleId`（唯一，kebab-case）**：`g4-experiment`
- **显示名称 `displayName`**：`4年级火车购票-交互`
- **版本号 `version`**：`1.0.0`
- **适用年级 / 学科 / 场景**：小学四年级 / 数学 / 场景：火车票购票
- **业务背景**：对小学4年级的科学探究能力和创新能力进行评估（前置实验/能力探测）
- **想达到的测评/教学目标**（四维度监测与评估）：
  - 问题识别（从情境对话中抽取“待解决问题”）
  - 信息提取与处理（从图表/地图/表格中提取信息并完成计算与筛选）
  - 想法生成（生成多种方案、提出改进方案）
  - 想法评估（判断方案优劣并给出理由，约束条件下做决策）
- **所属 Flow 类型**：前置实验步骤（实验）
  - Flow 结构说明：该子模块可作为 Flow 的前置实验模块运行；在 Flow 中的位置由 `FlowDefinition.steps[]` 决定
  - 默认总时长：40 分钟（见 5 节计时策略）

## 2. 运行环境与全局约束（Runtime & Global Constraints）[必填]

- **运行容器**：
  - 子模块 **必须** 运行在统一页面框架中：
    - 外壳：`AssessmentPageFrame`（导航 + 计时 + 主体 + 下一页按钮 + 错误托盘）
    - 左侧导航：`LeftStepperNav`
  - 子模块 **不得** 自行实现：全局导航、全局计时器、错误托盘、顶栏布局等
  - 说明：本子模块属于“历史模块包装”形态，业务页面复用 `src/modules/grade-4/*`；在 Flow 场景下通过 `VITE_USE_UNIFIED_FRAME=true` 切换为统一外壳渲染（见 10 节）

- **技术栈约束**：
  - 前端框架：React 18（JSX/TS 混用，新增代码优先 TypeScript）
  - 构建工具：Vite 4（由项目统一配置）
  - 风格规范：遵守项目现有 ESLint + Prettier 配置
  - 样式方案：CSS Modules（`*.module.css`）或与现有页面一致的样式组织

- **禁止直接操作的能力（必须通过统一封装）**：
  - 路由跳转：不得直接使用 `window.location`/`history.pushState` 或 Router API；**改用** 容器提供的 `onNext` / `onComplete` 等回调（或 `AssessmentPageFrame.onNext`）
  - 本地存储：不得随意读写 `localStorage/sessionStorage` 任意 key；**必须** 使用命名空间 `module.g4-experiment.*`（见 8 节）
  - 会话处理：不得自行处理 401 / session 过期；**必须** 交由统一 `handleSessionExpired`
  - 直接调用后端 API：除非明确允许，一般通过统一 API 层（如 `apiService`），并遵循数据格式规范

- **支持环境假设**：
  - 子模块可以假定：
    - 已存在 `userContext`（含 `examNo`、`batchCode`、`flowId` 等）
    - Flow 容器会传入 `initialPageId`，无需自行解析 URL
    - 全局错误捕获与统一提交管道已启用（`usePageSubmission`）

> **重要**：此表格应与 `mapping.ts` 中的 `PAGE_CONFIGS` 完全对应（见 6.1）。

### 4.2 页码映射规则（subPageNum ↔ pageId）

- `getInitialPage(subPageNum)` 实现规则：
  - 合法输入范围：`1..12`
  - 越界或缺失：回退到默认首页 `notices`
  - 实现方式：使用 `mapping.ts` 中的 `getPageIdBySubPageNum`
- 逆向映射（建议实现）：
  - `getSubPageNumByPageId(pageId)`：用于进度恢复与上报 `modulePageNum`

### 4.3 导航规则（前进/返回/跳转）

- **返回上一页**：默认不开放自由返回；按统一框架能力决定是否渲染“返回”入口。不得允许点击左侧圆点跳页。
- **点击“下一页”前置条件**（未满足则阻断并上报 `click_blocked`）：
  - `notices`：40s 倒计时结束 + 勾选“已阅读”
  - `problem-identification`：文本框非空
  - `factor-analysis`：至少选中 1 个选项
  - `route-analysis`：路线 1 与路线 5 距离均已填写且为合法数值
  - `station-recommendation`：已选择出发站 + 理由非空
  - `user-solution-design`：两种方案均至少放置 1 个任务块（总用时由系统计算并展示）
  - `plan-optimization`：必须选择“是/否”；若选“否”，需提供至少 1 个任务块的改进方案
  - `ticket-filter`：至少选择 1 条车次（建议同时高亮“是否符合约束”，但不强制只选符合项，具体按产品/测评口径）
  - `ticket-pricing`：推荐理由非空 + 票价计算过程已录入 + 总票价等于公式计算结果
- **特殊跳转**：
  - 全局计时到期：自动进入 `task-completion`（见 5.2 与 7.2）

## 5. 计时策略（Timing Strategy）[推荐]

### 5.1 默认计时配置（对应 `getDefaultTimers()`）

```ts
getDefaultTimers: () => ({
  task: 40 * 60, // 40 分钟（秒）
});
```

### 5.2 页面级倒计时与到期行为

- `notices` 页强制阅读倒计时：40 秒（模块内倒计时，不占用全局 task 计时）
  - 倒计时结束后才允许勾选确认
  - 记录事件：`timer_start` / `timer_complete`（或 `reading_complete`）
- 全局 task 计时（40 分钟）：
  - 建议从 `notices` 点击“下一页”进入 `scenario-intro` 时启动
  - 到期行为：
    - 触发自动提交（当前页未答题目填入 `"超时未回答"`）
    - 自动导航到 `task-completion`
    - 通知 Flow 容器 `flowContext.onTimeout?.()`（见 7.2）

### 6.3 MarkObject 数据格式规范（必须遵循）

- `pageNumber`：使用 `encodeCompositePageNum(stepIndex + 1, subPageNum)` 生成（格式 `X.YY`）
- `pageDesc`：使用 `buildPageDesc(PAGE_DESC_MAP[pageId], flowContext)` 生成（Flow 模式带前缀）
- `targetElement`：除保留元素外必须带前缀 `P{pageNumber}_`（用 `usePageMeta().prefixTarget` 或 `useOperationLogger` 自动加前缀）
- `beginTime/endTime`：统一格式 `YYYY-MM-DD HH:mm:ss`，且 `beginTime < endTime`

### 6.4 提交时机与失败处理

- **提交触发点**：
  - 点击“下一页/完成”时：收集当前页答案 → 补齐未答占位 → 构造 `MarkObject` → `usePageSubmission.submit()`；成功后才允许导航
  - 全局计时器到期：自动提交当前页 → 未答填 `"超时未回答"` → 导航到结束页
- **提交失败**：
  - 网络/服务端错误：错误托盘提示 + 阻断导航 + 允许重试（DEV 可配置 `allowProceedOnFailureInDev`，但必须在日志中标记）
  - 401/会话过期：交由 `handleSessionExpired` 统一处理

## 7. Flow 集成与进度恢复（Flow Integration）[必填]

### 7.1 进度恢复（modulePageNum → initialPageId）

- Flow 容器提供：`stepIndex`（当前 Flow 步骤索引）与 `modulePageNum`（子模块内页码）
- FlowModule 调用：`getInitialPage(modulePageNum)` 得到 `initialPageId`
- 子模块内部渲染对应页面并开始记录 `page_enter`（Flow 模式下自动注入 `flow_context`）

### 7.2 完成与超时判定

- **完成判定**：
  - 到达 `task-completion` 页并成功提交最后一页数据后，调用 `flowContext.onComplete?.()`
  - 注意：完成时不应再调用 `updateModuleProgress`，避免进度被错误回退
- **超时判定**：
  - 全局 task 计时到期：提交当前页数据并调用 `flowContext.onTimeout?.()`；随后进入 `task-completion`

### 7.3 Flow overrides 支持

- 支持 `options.timers` 覆盖默认计时（`FlowStep.overrides.timers` → `SubmoduleProps.options.timers`）
- 其他可扩展项（如 `variant`）：扩展前需先更新 `src/shared/types/flow.ts` 并在本说明书 11 节注明

## 9. 错误处理与边界情况（Error Handling & Edge Cases）[必填]

- **非法页码 / 配置错误**：
  - `modulePageNum` 越界或找不到 `pageId`：记录错误日志，回退到 `notices`
  - `mapping.ts` 配置缺失：视为不可恢复错误，应在开发/测试阶段通过 `validateMappingConfig` 发现
- **提交流程异常**：
  - 提交失败：展示错误托盘并阻断导航，允许重试
  - 401/会话过期：交由统一 `handleSessionExpired`
- **交互数据不完整**：
  - 拖拽方案为空、公式无法计算、输入非数字等：统一提示并阻断下一页，上报 `click_blocked`
- **全局计时器到期**：
  - 自动提交并进入结束页；未答题目按 `"超时未回答"` 填充

## 16. 最佳实践与反模式

### 16.1 最佳实践

| 实践 | 说明 |
|------|------|
| 使用 `collectAnswers` | 不自实现答案收集 |
| 使用 `useOperationLogger` | 自动处理序号、前缀与 flow_context 注入 |
| 使用 `encodeCompositePageNum` | 不硬编码 `pageNumber` |
| 使用 `buildPageDesc` | 统一生成 Flow 前缀 pageDesc |

### 16.2 常见反模式（避免）

| 反模式 | 风险 | 正确做法 |
|-------|------|----------|
| 硬编码 `pageNumber` | 易错且难维护 | `encodeCompositePageNum(stepIndex + 1, subPageNum)` |
| 手写 `flow_context` 注入 | 重复注入/跳码 | 用 `useOperationLogger` 或提交层自动注入 |
| 文本答案 JSON stringify 后带引号 | 影响后端解析与可读性 | `formatAnswer` 输出原始字符串 |
