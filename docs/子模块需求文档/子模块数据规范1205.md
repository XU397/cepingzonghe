## 1. 概述

### 1.1 设计原则

子模块数据提交遵循以下核心原则：

| 原则 | 描述 |
|------|------|
| **共享层负责"怎么提交"** | 统一的 pageNumber 规则、前缀规则、flow_context 注入、校验、实验历史等 |
| **子模块负责"提交什么"** | 页面/题目映射、页面状态、交互逻辑是业务私有的 |
| **约定优于配置** | 通过标准化的 mapping/config 接口接入，减少配置负担 |
| **增量迁移兼容** | 适配器模式，避免一次性重写 |

### 1.2 数据流概览

```
用户操作 → 子模块记录操作/答案 → 共享适配器格式化 → 提交到后端
    ↓              ↓                    ↓
  logOperation   collectAnswers      buildMark
  collectAnswer  appendExperimentHistory  validateMark
```

```typescript
  // ...

// 6. 答案键到问题ID的映射
export const ANSWER_KEY_TO_QUESTION: Record<string, string> = {
  controlVariableReason: 'controlVariableReason',
  minGsdFocal: 'minGsdFocal',
  // ...

// 7. 子页码查询函数（必需）
export function getSubPageNumByPageId(pageId: string): number {
  const config = PAGE_CONFIGS.find(c => c.pageId === pageId);
  return config?.subPageNum ?? 1;
}

// 8. 页面ID查询函数
export function getPageIdBySubPageNum(subPageNum: number): string | undefined {
  const config = PAGE_CONFIGS.find(c => c.subPageNum === subPageNum);
  return config?.pageId;
}
```

## 4. 数据收集规范

### 4.1 操作记录（operationList）

#### 4.1.1 Operation 结构

```typescript
interface Operation {
  code: number;           // 序号，从 1 开始递增
  targetElement: string;  // 目标元素，需带前缀
  eventType: string;      // 事件类型
  value: string;          // 事件值
  time: string;           // 时间戳，格式：YYYY-MM-DD HH:mm:ss
}
```

#### 4.1.2 事件类型规范

| 事件类型 | 说明 | value 格式 |
|----------|------|------------|
| `page_enter` | 进入页面 | 页面ID |
| `page_exit` | 离开页面 | 页面ID |
| `click` | 点击 | 点击的内容或空 |
| `input` | 文本输入 | `{"prev": "旧值", "next": "新值"}` |
| `select` | 选择 | 选中的值 |
| `flow_context` | Flow 上下文 | FlowContext JSON |
| `experiment_run` | 实验运行 | 实验参数 JSON |
| `experiment_capture` | 实验捕获 | 捕获数据 JSON |

#### 4.1.3 targetElement 前缀规则

**格式：** `P{stepIndex+1}.{subPageNum:02d}_{元素描述}`

**示例：**
- `P1.01_下一步按钮` → 第1步第1页的下一步按钮
- `P2.03_原因输入框` → 第2步第3页的原因输入框
- `P1.04_实验数据捕获` → 第1步第4页的实验捕获
- `P1.02_控制变量选择` → 第1步第2页的控制变量选择
- `P1.05_焦距滑块` → 第1步第5页的焦距滑块

**保留元素（不加前缀）：**

| 中文名称 | 英文常量 | 说明 |
|----------|----------|------|
| `页面` | `page` | 页面级事件（page_enter/page_exit） |
| `下一步按钮` | `next_button` | 导航按钮 |
| `上一步按钮` | `prev_button` | 导航按钮 |
| `flow_context` | `flow_context` | Flow 上下文（系统字段，保持英文） |
| `任务计时器` | `task_timer` | 任务计时器 |
| `倒计时` | `countdown` | 倒计时 |
| `模块` | `module` | 模块级事件 |

> **工具函数**：使用 `isReservedElement(element)` 判断是否为保留元素，或查询 `RESERVED_ELEMENTS` 常量集合。

#### 4.1.4 必需事件

每个页面必须包含：
1. `page_enter` - 进入页面事件（第一个事件）
2. `page_exit` - 离开页面事件（最后一个事件）
3. `flow_context` - Flow 上下文事件（Flow 模式下必需，紧跟 page_enter）

### 4.2 答案记录（answerList）

#### 4.2.1 AnswerEntry 结构

```typescript
interface AnswerEntry {
  code: number;           // 问题编码，对应 QUESTION_CODE_MAP
  targetElement: string;  // 问题完整文本，对应 QUESTION_TEXT_MAP
  value: string;          // 答案内容（JSON 字符串或纯文本）
}
```

#### 4.2.2 答案收集流程

```typescript

// 收集当前页面的答案
const answerList = collectAnswers(currentPageId, answers, {
  PAGE_CONFIGS,
  PAGE_DESC_MAP,
  PAGE_QUESTIONS,
  QUESTION_CODE_MAP,
  QUESTION_TEXT_MAP,
  ANSWER_KEY_TO_QUESTION,
  getSubPageNumByPageId,
  QUESTION_OPTIONS_MAP,  // 可选：用于格式化答案
});
```

#### 4.2.3 实验历史追加

```typescript
import { appendExperimentHistory } from '@shared/services/submission/submoduleAdapter';

// 追加实验历史数据
if (hasExperimentData) {
  answerList = appendExperimentHistory(answerList, experimentData, {
    targetElement: `P${pageNumber}_实验数据记录`,
```

## 5. 数据提交规范

### 5.1 MarkObject 结构

```typescript
interface MarkObject {
  pageNumber: string;       // 页码，格式：{stepIndex+1}.{subPageNum:02d}
  pageDesc: string;         // 页面描述，Flow 模式下带前缀
  operationList: Operation[];
  answerList: AnswerEntry[];
  beginTime: string;        // 进入时间，格式：YYYY-MM-DD HH:mm:ss
  endTime: string;          // 离开时间，格式：YYYY-MM-DD HH:mm:ss
  imgList: string[];        // 图片列表（可选）
}
```

### 5.2 pageNumber 规则

**格式：** `{stepIndex + 1}.{subPageNum:02d}`

| stepIndex | subPageNum | pageNumber |
|-----------|------------|------------|
| 0 | 1 | `1.01` |
| 0 | 3 | `1.03` |
| 1 | 2 | `2.02` |
| 2 | 5 | `3.05` |

```typescript
import { encodeCompositePageNum } from '@shared/utils/pageMapping';

const pageNumber = encodeCompositePageNum(stepIndex + 1, subPageNum);
// encodeCompositePageNum(1, 3) → "1.03"
```

### 5.3 pageDesc 规则

**普通模式：**
```
假设问题
```

**Flow 模式：**
```
[flow-01/g8-drone/2] 假设问题
```

```typescript
import { buildPageDesc } from '@shared/services/submission/submoduleAdapter';

const pageDesc = buildPageDesc(
  PAGE_DESC_MAP[currentPageId],
  flowContext  // null 或 FlowContext 对象
);
```

### 5.4 时间格式

**格式：** `YYYY-MM-DD HH:mm:ss`

```typescript
import { formatTimestamp } from '@shared/services/dataLogger';

const time = formatTimestamp(new Date());
// "2025-12-05 14:30:45"
```

### 5.5 Flow Context 结构

```typescript
interface FlowContext {
  // 核心字段
  flowId: string;        // Flow 唯一标识
  submoduleId: string;   // 子模块 ID
  stepIndex: number;     // 当前步骤索引
  moduleName?: string;   // 子模块显示名称（推荐填写，为兼容旧代码设为可选）
  pageId?: string;       // 当前页面 ID

  // 可选扩展字段（供 Flow 运行态使用，提交时不要求）
  totalSteps?: number;   // 总步骤数
  flowStartTime?: string; // Flow 开始时间
}
```

> **注意**：`totalSteps` 和 `flowStartTime` 为 Flow 运行态扩展字段，提交层序列化时不强制要求，接入方可按需使用。

### 5.6 提交前验证

```typescript
import { validateMarkBeforeSubmit } from '@shared/services/submission/submoduleAdapter';

const result = validateMarkBeforeSubmit(mark, flowContext);

if (!result.valid) {
  console.error('提交验证失败:', result.errors);
  // 错误类型：
  // - INVALID_PAGE_NUMBER: 页码格式错误
  // - MISSING_PREFIX: 缺少前缀
  // - CODE_DISCONTINUITY: 序号不连续
  // - MISSING_REQUIRED_EVENT: 缺少必需事件
  // - MISSING_FLOW_CONTEXT: 缺少 flow_context
  // - INVALID_TIME_FORMAT: 时间格式错误
  // - INVALID_TIME_RANGE: 时间范围错误
}
```

### 5.7 输出字段顺序要求

标记数据序列化输出时（例如提交前的 JSON），字段顺序必须保持以下顺序，便于一致性审计与外部工具读取：
1) `pageNumber`
2) `pageDesc`
3) `operationList`
4) `answerList`
5) `beginTime`
6) `endTime`
7) `imgList`

> 说明：这是输出序列化时的字段顺序要求，字段内容及验证规则保持不变，所有字段需始终输出（空集合也输出为空数组），适用于提交 payload、日志导出、快照/调试输出等所有 MarkObject 序列化场景。

```typescript
const { pageNumber, targetPrefix, prefixTarget } = usePageMeta(stepIndex, subPageNum);
```

### 6.4 完整提交示例

```typescript
const handleSubmit = async () => {
  const subPageNum = getSubPageNumByPageId(currentPageId);
  const pageNumber = encodeCompositePageNum(stepIndex + 1, subPageNum);

  // 1. 收集答案
  let answerList = collectAnswers(currentPageId, answers, SUBMODULE_MAPPING_CONFIG);

  // 2. 追加实验历史（如有）
  if (hasExperimentData) {
    answerList = appendExperimentHistory(answerList, experimentData, {
      targetElement: `P${pageNumber}_实验数据记录`,
      historyCodeBase: HISTORY_CODE_BASE,
    });
  }

  // 3. 构建 Mark 对象
  const mark: MarkObject = {
    pageNumber,
    pageDesc: buildPageDesc(PAGE_DESC_MAP[currentPageId], flowContext),
    operationList: operations,
    answerList,
    beginTime: formatTimestamp(pageEnterTime),
    endTime: formatTimestamp(new Date()),
    imgList: [],
  };

  // 4. 验证
  const validation = validateMarkBeforeSubmit(mark, flowContext);
  if (!validation.valid) {
    console.error('验证失败:', validation.errors);
    return;
  }

  // 5. 提交
  await submit(mark);
};
```

## 7. 时间管理规范

### 7.1 核心原则

**时间更新必须与页面切换同步，不能使用 useEffect 异步更新。**

### 7.2 正确实现（Reducer 模式）

```typescript
// 在 reducer 中同步更新
case 'SET_PAGE':
  return {
    ...state,
    currentPageId: action.payload,
    pageEnterTime: new Date(),  // 原子更新
    operations: [],
    operationSequence: 1,
    flowContextInjected: false,
  };
```

### 7.3 错误实现（useEffect 模式）

```typescript
// ❌ 不要这样做
useEffect(() => {
  setPageEnterTime(new Date());  // 异步，有竞态风险
}, [currentPageId]);
```

### 7.4 时间记录要求

| 时间字段 | 记录时机 | 说明 |
|----------|----------|------|
| `beginTime` | 页面切换时 | 与 currentPageId 同步更新 |
| `endTime` | 提交时 | `new Date()` 当前时间 |
| `operation.time` | 操作发生时 | 每次操作的实际时间 |

### 8.3 子模块合规检查清单

#### 8.3.1 Mapping 配置合规

| 检查项 | 要求 | 验证方式 |
|--------|------|----------|
| PAGE_CONFIGS | 必须定义，数组格式 | `validateMappingConfig` |
| PAGE_DESC_MAP | 所有 pageId 都有描述 | 单元测试 |
| PAGE_QUESTIONS | 定义每个页面的问题键 | 单元测试 |
| QUESTION_CODE_MAP | 所有问题都有编码 | 单元测试 |
| QUESTION_TEXT_MAP | 所有问题都有完整文本 | 单元测试 |
| ANSWER_KEY_TO_QUESTION | 答案键到问题 ID 映射完整 | 单元测试 |
| getSubPageNumByPageId | 必须导出此函数 | 编译检查 |

#### 8.3.2 数据格式合规

- [ ] pageNumber 格式正确（X.YY，如 `1.03`）
- [ ] targetElement 前缀正确（如 `P1.03_下一步按钮`）
- [ ] code 从 1 开始连续递增，无跳跃
- [ ] 包含 page_enter 事件（第一个事件）
- [ ] 包含 page_exit 事件（最后一个事件）
- [ ] Flow 模式下包含 flow_context 事件（紧跟 page_enter）
- [ ] beginTime < endTime（进入时间早于离开时间）
- [ ] 时间格式正确（YYYY-MM-DD HH:mm:ss）
- [ ] answerList 中的 code 对应 QUESTION_CODE_MAP
- [ ] answerList 中的 targetElement 对应 QUESTION_TEXT_MAP

#### 8.3.3 操作管理合规

| 检查项 | 要求 | 说明 |
|--------|------|------|
| 序号管理 | 页面切换时重置为 1 | 避免跨页序号累加 |
| flow_context 注入 | 每个新页面注入一次 | 紧跟 page_enter 事件 |
| 注入标记重置 | 页面切换时重置 flowContextInjected | 确保每页都能注入 |
| Utility 使用 | 使用共享 utility 管理序号与注入 | 避免手写逻辑导致遗漏 |

**Utility 使用说明：**
- 新模块优先使用 `useOperationLogger` 全托管模式，序号管理、flow_context 注入与页面状态重置均由共享 utility 处理。
- 兼容模式下，必须使用 `createOperationSequence` 生成序号、`injectFlowContext` 注入上下文，切页时通过 `createPageStateResetter` 重置序号与注入标记。
- 禁止自实现递增或手动插入 flow_context，避免重复注入或跳码。

#### 8.3.4 时间管理合规

| 检查项 | 要求 | 反模式 |
|--------|------|--------|
| pageEnterTime 更新时机 | 与 pageId 变化同步 | ❌ useEffect 异步更新 |
| pageEnterTime 存储方式 | Reducer 状态或同步 Ref | ❌ 依赖 AppContext 异步传递 |
| beginTime 获取 | 从状态/Ref 直接读取 | ❌ useMemo 闭包捕获 |

**推荐模式：**
```typescript
// Reducer 模式（推荐）
case 'SET_PAGE':
  return {
    ...state,
    currentPageId: action.payload,
    pageEnterTime: new Date(),  // 同步更新
  };

// 或 Ref 同步模式
if (currentPageId !== lastPageIdRef.current) {
  pageEnterTimeRef.current = new Date();
  lastPageIdRef.current = currentPageId;
}
```

## 10. 操作管理 Utility

### 10.1 概述

操作管理 Utility 提供统一的序号管理和 flow_context 注入功能，解决以下问题：
- 序号管理分散实现导致的 bug（如 SET_PAGE 未重置序号）
- flow_context 注入时机不一致
- 规范要求的重复实现

### 10.2 createOperationSequence

轻量的操作序号管理器，用于生成单页内连续的 code 序列。

**设计决策：**
- 使用闭包实现，保持轻量
- 固定从 0 开始，next() 首次返回 1（符合 code 从 1 开始的规范）
- 不提供 initialValue 参数：断点续做恢复的是页面位置，不是操作序号

### 10.3 injectFlowContext

在操作列表中注入 flow_context 事件，保持 page_enter 之后的顺序。

**注入规则：**
- 必须在 page_enter 事件之后注入
- 缺失 page_enter 时抛出 MissingPageEnterError
- 使用 shouldInjectFlowContext 检查，避免重复注入

### 10.4 createPageStateResetter

生成切页重置器，重置操作序号并可选重置 flow_context 注入标记，防止跨页复用状态。

### 10.5 子模块接入模式

#### 模式 A：全托管（推荐新模块）

直接使用 useOperationLogger，内部已集成所有 utility：
- 自动使用 createOperationSequence 管理 code
- page_enter 后按需自动 injectFlowContext
- 切页时通过 createPageStateResetter 重置序号与注入标记

#### 模式 B：灵活组合（兼容老模块）

在 Reducer 中使用独立 utility：
- Reducer 初始化 `const sequence = createOperationSequence();`
- 记录 page_enter 后，用 `injectFlowContext` 注入 flow_context（前置 shouldInjectFlowContext 判断）
- 页面切换时调用 `createPageStateResetter(sequence, setFlowContextInjected)()` 重置序号与注入状态

### 10.6 导出清单

| 导出项 | 类型 | 说明 |
|--------|------|------|
| createOperationSequence | function | 创建序号管理器 |
| OperationSequence | interface | 序号管理器接口 |
| injectFlowContext | function | 注入 flow_context 事件 |
| shouldInjectFlowContext | function | 检查是否需要注入 |
| FlowContextPayload | interface | flow_context 数据结构 |
| InjectFlowContextOptions | interface | 注入选项 |
| MissingPageEnterError | class | 缺失 page_enter 错误 |
| createPageStateResetter | function | 创建页面状态重置器 |
| PageStateResetOptions | interface | 重置选项 |
| RESERVED_ELEMENTS | Set\<string\> | 保留元素集合（不加前缀的元素） |
| isReservedElement | function | 判断元素是否为保留元素 |

### 10.7 提交层重复注入检测

提交层（usePageSubmission）在自动注入 `flow_context` 前，**必须**检测 `operationList` 中是否已存在 `eventType: 'flow_context'` 的事件：

```typescript
// 提交层逻辑
if (shouldInjectFlowContext(operationList, flowContext)) {
  operationList = injectFlowContext(operationList, flowContext, sequence);
}
// 若已存在 flow_context 事件，跳过注入
```

此机制确保：
- **全托管模式**：useOperationLogger 已自动注入，提交层不重复注入
- **灵活组合模式**：子模块手动注入后，提交层不重复注入
- **兼容性**：无论哪种模式，最终提交数据中只有一个 flow_context 事件
