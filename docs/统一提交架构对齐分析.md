# 统一提交架构对齐分析

## 背景
- 统一提交新架构：组件通过 `submission={ { operations, answers, getUserContext, getFlowContext } }` 交付，由 `usePageSubmission` 组装 mark。
- 老架构：依赖 AppContext 的 `preparePageSubmissionData()`（`pageInfoMapping` 等全局数据）生成 mark。
- 框架适配层：`AssessmentPageFrame` 包装 `buildMark` 并把它传给 `usePageSubmission`，兼容旧模块。

## 现状
- 框架逻辑：当 `submission.buildMark` 缺省时，`AssessmentPageFrame` 兜底为 `buildMarkFromContext`（`src/shared/ui/PageFrame/AssessmentPageFrame.jsx:388`）。
- Hook 优先级：`usePageSubmission` 只要收到 `buildMark` 就不会走 `operations/answers` 兜底（`src/shared/services/submission/usePageSubmission.js:515`）。
- g8-pv-sand-experiment：仅提供 `operations/answers`，未提供 `buildMark`，结果被框架兜底拉回 AppContext，`instructions_cover` 未在 `pageInfoMapping` 注册而报错。
- g8-mikania-experiment / g8-drone-imaging：各自实现 `buildMark`，不会触发框架兜底，因此正常。

## 问题
- 兜底范围过宽：即便模块已提供 `operations/answers`（新架构），仍被强制走老路径。
- 结果：`operations/answers` 被忽略，落入 `preparePageSubmissionData()`，而新子模块的 `pageId` 未在全局映射，导致 “Cannot find page details for …”。
- 架构偏移：新架构的自动注入/校验能力（flow_context、schema 等）被绕过。

## 对齐方案（推荐）
- 调整 `AssessmentPageFrame` 的 `wrappedBuildMark`：
  - 有 `submission.buildMark` → 按原样使用（兼容旧模块）。
  - 无 `buildMark` 但有 `operations` 或 `answers` → 返回 `undefined`，让 `usePageSubmission` 使用新路径构建 mark。
  - 仅在既无 `buildMark` 又无 `operations/answers` 时，才回退 `buildMarkFromContext`。
- 影响面：
  - 老模块：无 `operations/answers` 的仍走 AppContext，不变。
  - 新模块：提供 `operations/answers` 的转向新路径，避免依赖 `pageInfoMapping`。
  - 已自带 `buildMark` 的模块（如 g8-mikania、g8-drone-imaging）行为不变。

## 验证建议
- 针对 Frame 改动，回归一次统一提交相关用例（含超时、重试、next_click 注入）。
- 覆盖提供 `buildMark`、仅提供 `operations/answers`、都未提供三种配置，确认分支符合预期。
- 关注 `appendNextClickEvent`、生命周期 `operationList` 的合并仍符合 schema。

## 后续思考
- 逐步引导仍在用 `buildMark` 的模块迁移到 `operations/answers`，享受自动注入/校验能力（可分批，不阻塞本次修复）。
- 将新模块 `pageId` 补齐到 `pageInfoMapping` 仅作为兜底，不作为主要路径。***
