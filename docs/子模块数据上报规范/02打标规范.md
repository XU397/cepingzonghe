# 02 打标规范

适用对象：后端基于 `01轨迹上报规范.md` 中的标准轨迹事件，生成香蕉变黑及同构科学探究任务的行为标签、策略特征和可供 HMM / Viterbi / 后续认知分析使用的结构化结果。同构任务指页面结构一致、仅主题内容不同的子模块，例如蜂蜜黏度、蒸馒头等。

本文档面向 AI 编码工程师和 Java 后端工程师，目标是使 Java 类算法能按统一输入、统一规则、统一输出完成打标。

版本状态（2026-06-06）：本文正文保留 v2.1 系列打标口径和部分示例；当前香蕉变黑前端运行时已经升级到 v2.2，前后端联调和新验收应以 `engineering_contracts/*.v2.2.json`、`acceptance_cases` 中的 v2.2 metadata 以及 `docs/superpowers/specs/2026-06-06-banana-trace-v2-2-frontend-implementation-design.md` 为准。v2.1 文件作为历史兼容补丁镜像保留，其中 `event_schema.v2.1.json` 已回补 `PAGE_IDLE` 以保证同版本 docs/runtime 副本一致。

## 1. 总体原则

### 1.1 分层边界

| 层级 | 责任 | 是否由本文档定义 |
|---|---|---:|
| L0 原始日志层 | 前端记录学生客观操作 | 由 `01轨迹上报规范.md` 定义 |
| L1 控件字典与 UI 约束层 | 页面、控件、字段、题号、选项、行 ID | 由两个文档共同约束 |
| L2 标准事件层 | `START_PAGE`、`TEXT_CHANGE`、`EXECUTE_EXP` 等 | 由 `01轨迹上报规范.md` 定义 |
| L3 行为标签层 | `GENERATE_THINK`、`REVISE`、`CHECK_REVIEW` 等 | 由本文档定义 |
| L4 结果 Metadata 层 | 次数、时长、字符变化、策略特征 | 本阶段暂不实现，仅保留边界说明和后续扩展口径 |
| L5+ 模型推断层 | HMM、Viterbi、能力指标、学生报告 | 不在本文档直接实现 |

核心原则：

- 前端只记录事实，后端生成标签。
- 打标只生成过程证据，不直接判定学生能力高低。
- 能力维度优先跟页面任务目标对齐，操作标签作为过程证据。
- 所有阈值进入 `RuleConfig`，不得硬编码在规则内部。
- 打标算法按页面拓扑和字段角色复用，不得写死香蕉、蜂蜜、馒头等具体主题词；主题展示名只能出现在 Field Registry 的 module/display config 或 Content Registry 中，不得进入标准 `field_id` 或算法代码。
- 本阶段后端只实现 L2 标准事件到 L3 行为标签，以及必要的数据质量校验；L4 特征聚合、评价生成和能力推断后续单独立项。
- 页面生命周期事件（`START_PAGE`、成功 `SUBMIT_ATTEMPT`、`TASK_FINISH`）只进入 `pageMetadata.lifecycle`，不进入 L3 `labels`、`sequence_json`、HMM / Viterbi。
- `PAGE_IDLE` 是 L2 页面状态事件，默认不直接生成 L3 行为标签，也不直接进入 HMM / Viterbi；后端可将其写入 `pageMetadata.idleSegments`、`strategyFlags` 或后续 L4 特征，用作页面定向、读说明、思考、犹豫、补救前停顿等候选证据。

### 1.1.1 L2 / L3 / L4 命名边界

后端实现必须区分三类对象：

| 层级 | 来源 | 示例 | 是否默认进入 HMM |
|---|---|---|---:|
| L2 standard event | 前端 `operationList[*].eventType` | `START_PAGE`、`SUBMIT_ATTEMPT`、`PAGE_IDLE`、`CHAT_SCROLL`、`TEXT_CHANGE` | 否 |
| 页面生命周期 / 流程上下文 | 后端从 L2 生命周期事件汇总 | `pageMetadata.lifecycle.startPage`、`submitSuccess`、`taskFinish` | 否 |
| L3 behavior label | 后端规则生成 | `READ_CHAT`、`REVIEW_CHAT`、`SELECT_OPTION`、`GENERATE_THINK` | 是，除非标签显式 `useInHmm=false` |
| L4 metadata / page state | 后端聚合特征 | `FIELD_COMPLETENESS`、`reading_features`、`key_evidence_coverage_ratio` | 否；本阶段不要求输出 |

`supportedLabels` 只指 L3 行为标签，不包含 L2 standard event，也不包含 `START_PAGE`、`SUBMIT_PAGE`、`TASK_FINISH` 等页面生命周期边界。若某个 L3 标签由同名 L2 事件直接映射，必须标记 `labelSource=direct_mapping` 和 `useInHmm` 取值，避免工程事件未经解释直接进入 HMM 序列。

### 1.2 页面主能力维度

| 页面 | 主能力维度 | 说明 |
|---|---|---|
| 第 1 页 | 非能力流程页 | 背景页 |
| 第 2 页 | 信息筛选与整合能力 | 基于聊天情境提出科学问题 |
| 第 3 页 | 信息筛选与整合能力 | 基于资料卡进行多因素判断 |
| 第 4 页 | 非能力流程页 | 阶段过渡 |
| 第 5 页 | 探究方案设计能力 | 提出三个方案 / 想法 |
| 第 6 页 | 探究方案评估能力 | 评价不同方案优缺点 |
| 第 7 页 | 非能力流程页 | 实验阶段过渡 |
| 第 8 页 | 数据读取与分析能力 | 是否使用模拟实验工具获取证据 |
| 第 9 页 | 数据读取与分析能力 | 实验 + 题组第 1 题 |
| 第 10 页 | 数据读取与分析能力 | 实验 + 题组第 2 题 |
| 第 11 页 | 数据读取与分析能力 | 实验 + 题组第 3 题 |
| 第 12 页 | 探究方案评估能力 | 综合页；允许 `CHART_INSPECT` 辅助映射到数据读取与分析能力 |
| 第 13 页 | 非能力流程页 | 结束页 |

第 12 页例外：它是综合页，`CHART_INSPECT` 可辅助映射到数据读取与分析能力；理由文本质量、论证质量和答案正确性由评分 / 评价模块处理，不在本阶段输出 L3 `ARGUMENTATION`。

### 1.3 标签生成顺序

同一页面内建议按以下顺序执行：

1. 事件标准化与排序。
2. 噪音过滤与重复事件合并。
3. 页面生命周期汇总：`START_PAGE`、成功 `SUBMIT_ATTEMPT`、`TASK_FINISH` 写入 `pageMetadata.lifecycle`。
4. L3 受阻 / 补救标签：`INVALID_SUBMIT`、`REVIEW_AFTER_BLOCKED`。
5. 控件级过程标签：文本、资料卡、选择题、实验、图表、方案表。
6. 跨事件标签：`REVISE`、`REVISE_ANSWER`、`CHECK_REVIEW`、`CROSS_FIELD_REVIEW`。
7. 页面级 Metadata 汇总。
8. 策略特征生成。
9. 验证不得出现的标签。

标签输出补充规则：

- `labels` 允许重复；多次阅读、输入、修改、实验、选择、回看只要满足有效阈值，都应按证据顺序生成重复 L3 标签。
- `labels` 输出顺序按 `evidenceStartEventSeq`、`evidenceStartTimeMs`、`generationOrder` 排序。
- 不做标签层去重，不做连续重复压缩。
- `sequence_json` 必须由 `detail_json.labels` 投影生成，保留重复与真实顺序。
- `sequence_json` 只包含 L3 核心行为标签；页面只有生命周期事件时，`labels` 与 `sequence_json` 可以为空。
- 后端 Normalizer 只去工程噪声，不做认知语义压缩，不因标签重复而删除事件。

### 1.4 标签 / 标记归属表

| 对象 | 进入 `detail_json.labels` | 进入 `sequence_json` / HMM | 说明 |
|---|---:|---:|---|
| 学生可观察 L3 行为标签，且 `useInHmm=true` | 是 | 是 | 例如 `READ_CHAT`、`WRITE_REASON`、`CHECK_REVIEW`、`INVALID_SUBMIT` |
| 页面生命周期事件 | 否 | 否 | `START_PAGE`、成功 `SUBMIT_ATTEMPT`、`TASK_FINISH` 只进入 `pageMetadata.lifecycle` |
| `strategyFlags` | 否 | 否 | 页面级过程模式线索，只进入 `detail_json.strategyFlags` |
| `pageMetadata` | 否 | 否 | 页面级统计、状态或 L4 future 候选，只进入 `detail_json.pageMetadata` |
| `dataQualityWarnings` | 否 | 否 | 数据质量审计信息，只进入 `detail_json.dataQualityWarnings` |
| `suppressedEvents` | 否 | 否 | 工程噪声过滤 / 合并明细，只进入 `detail_json.suppressedEvents` |
| `FIELD_COMPLETENESS` | 否 | 否 | 当前只作为 pageMetadata / L4 future 候选 |
| `ARGUMENTATION` | 否 | 否 | 当前不输出；后续如需要文本表层线索，应放 L4 future 或评分模块且 `useInHmm=false` |

## 2. 输入输出 Schema

### 2.1 输入：页面批量 Mark JSON + 内部 TraceEvent

前端输入保持页面批量 Mark JSON 总体结构：`pageNumber`、`pageDesc`、`operationList`、`answerList`、`beginTime`、`endTime`、`imgList`。后端只把 `operationList[*]` 解析为内部 `TraceEvent`；`answerList` 保留给评分模块，不进入轨迹打标。

Java 类内部应按以下字段解析标准化事件：

```java
public class TraceEvent {
    private String traceId;
    private String sessionId;
    private String studentId;
    private String taskId;
    private int pageIndex;
    private String pageId;
    private String pageType;
    private String fieldRegistryVersion;
    private String fieldRegistryHash;
    private long eventSeq;
    private String eventType;
    private String targetId;
    private String targetType;
    private String fieldId;
    private String contentId;
    private String questionId;
    private String optionId;
    private String rowId;
    private OffsetDateTime clientTime;
    private Object valueBefore;
    private Object valueAfter;
    private String validationStatus;
    private Map<String, Object> metadata;
}
```

实现要求：

- 优先按 `event_seq` 排序；若缺失再按 `client_time` 排序。
- 若 `event_seq` 重复，按 `client_time` 和输入顺序兜底，但必须记录数据质量告警。
- `client_time` 必须按毫秒级解析；标准格式示例为 `2026-06-03T10:30:10.123+08:00`。
- 旧 Mark JSON 的秒级 `time` 不得被伪造为毫秒级，只能标记 `metadata.time_precision="second"` 并降低停顿、快速切换、重复点击等规则的可信度。
- `metadata` 缺失时按空 Map 处理。
- 新数据中，`field_id`、`content_id`、`question_id`、`option_id`、`row_id` 应由 `operationList[*].value` 结构化提供；旧 Mark JSON 不做历史迁移，`TraceEventNormalizer` 不承担旧 `targetElement` 文案解析职责。
- 页面批量上报时，`field_registry_version` 与 `field_registry_hash` 由 `START_PAGE.value.metadata` 承载，后端解析为 `TraceEvent` 时注入当前页上下文；若未来采用流式事件上报，单事件必须携带这两个字段或可解析到等价 session context。
- `fieldRegistryVersion` 用于选择静态字段注册表；后端不得使用前端动态传入的字段语义覆盖注册表；`contentId` 的语义由 Content Registry 解释。
- 任何规则不得依赖前端传来的能力判断字段。

### 2.2 输出：TraceLabel

```java
public class TraceLabel {
    private String code;
    private boolean useInHmm;
    private List<String> evidenceEventIds;
    private Long evidenceStartEventSeq;
    private long evidenceStartTimeMs;
    private long startTimeMs;
    private long endTimeMs;
    private String abilityDimension;
    private double confidence;
    private Map<String, Object> metadata;
}
```

输出要求：

- `evidenceEventIds` 必须可追溯到输入事件。
- `confidence` 默认规则标签为 `1.0`；弱证据标签可低于 `1.0`。
- 内部时间字段统一为 epoch millis long，字段名使用 `xxxTimeMs`。
- 标签输出顺序按 `evidenceStartEventSeq`、`evidenceStartTimeMs`、`generationOrder` 排序。
- 同一 L3 标签允许重复输出；同一标签如由多个事件合并生成，`metadata` 应保留计数、时长、目标 ID 列表。
- `metadata` 只允许保存 ID、计数、时长、布尔值、枚举、hash、证据引用等安全字段，不得保存学生原文、文本摘要或大段内容。

### 2.3 页面级结果：PageTagResult

```java
public class PageTagResult {
    private String schemaVersion;
    private String executionStatus; // SUCCESS / FAILED
    private String fatalReasonCode;
    private String fatalReasonMessage;
    private String algorithmId;
    private String algorithmVersion;
    private String ruleConfigVersion;
    private String ruleConfigHash;
    private String fieldRegistryVersion;
    private String fieldRegistryHash;
    private String contentRegistryVersion;
    private String contentRegistryHash;
    private String sourceTraceHash;
    private long generatedAtMs;
    private List<TraceLabel> labels;
    private Map<String, Object> pageMetadata;
    private List<String> strategyFlags;
    private List<DataQualityWarning> dataQualityWarnings;
    private List<SuppressedEvent> suppressedEvents;
}
```

本阶段输出边界：

- `labels` 是本阶段核心交付，只输出 `useInHmm=true` 的 L3 可观察行为标签；页面只有生命周期事件时允许为空。
- `sequence_json` 必须从 `PageTagResult.labels` 投影生成，不允许独立生成，也不得包含 `START_PAGE`、`SUBMIT_PAGE`、`TASK_FINISH`。
- `detail_json` 保存完整 `PageTagResult`，但不得保存完整原始 `operationList`、学生原文、文本摘要或大段内容。
- `sourceTraceHash` 对原始 `operationList` 的 canonical JSON 计算，用于审计追溯，不替代原始轨迹存储。
- `generatedAtMs` 为生成结果时的 epoch millis，与其他内部时间字段保持 `xxxTimeMs` 命名一致。
- `dataQualityWarnings` 是本阶段必交付，每项必须包含 `severity=FATAL|WARN`，用于前后端联调和上报质量治理。
- `pageMetadata` 仅允许放解析辅助信息，例如 `page_duration_ms`、`effective_page_duration_ms`、`registry_checked=true`，以及 `lifecycle.startPage`、`lifecycle.submitSuccess`、`lifecycle.taskFinish` 等页面生命周期状态；不得在本阶段扩展为完整 L4 特征集。
- `strategyFlags` 保留页面级过程模式标记，例如第 8 页 `NO_EXP_DIRECT_NEXT`，不得解释为能力结论，也不进入 `labels` / `sequence_json` / HMM；Viterbi 后聚合可与 L3 路径共同使用。
- `suppressedEvents` 记录被过滤或合并的工程噪声事件明细；不得静默删除事件。

### 3.2 接口

```java
public interface TraceTaggingService {
    PageTagResult tag(String markJson, String algorithmId);
}

public interface ITraceTagAlgorithm {
    String algorithmId();
    boolean supports(String pageId, String pageType);
    PageTagResult tag(NormalizedTracePage page, TaggingContext context);
}
```

打标入口输入边界：

- 对外入口是 `TraceTaggingService`，接收完整 DB `mark` JSON 和 pipeline 配置快照中的 `algorithmId`。
- `ITraceTagAlgorithm` 不接收原始 Mark JSON，只接收 `NormalizedTracePage + TaggingContext`。
- `TraceEventNormalizer` 只接收新标准轨迹事件序列，不接收旧 Mark JSON 历史数据。
- `algorithmId` 由后端 pipeline / registry 配置校验后选择；学生前端事件不能直接决定算法。
- `FieldRegistry` 解释所有会被打标消费的 `field_id`；字段语义不从事件流动态读取。
- `answerList` 必须保留给评分 / 答案判定模块使用，但轨迹打标算法不得读取或分析 `answerList`。
- `TEXT_BLUR.value_after` 只能作为行为过程证据，不替代 `answerList` 成为评分答案源。

`SequenceProjector` 规则：

```text
sequence_json = detail_json.labels
  .filter(label.useInHmm == true)
  .sortBy(evidenceStartEventSeq, evidenceStartTimeMs, generationOrder)
  .map(label.code)
```

- 保留重复标签。
- 不按固定优先级重新排序。
- 不去重，不压缩连续重复。
- `strategyFlags`、`dataQualityWarnings`、`suppressedEvents`、`pageMetadata` 不参与投影。

### 3.3 FieldRegistry 与 PageTopologyResolver

Field Registry 是前后端共享的静态字段注册表，不来自事件流。Java 打标入口必须先完成：

1. 从 `START_PAGE.value.metadata` 读取 `field_registry_version/hash` 和 `content_registry_version/hash`。
2. 根据 `fieldRegistryVersion` 加载对应 registry。
3. 校验 `fieldRegistryHash`，若上报了 hash 且不一致，输出 `FIELD_REGISTRY_VERSION_MISMATCH`。
4. 根据 `page_id + field_id` 解析字段定义。
5. 由 `PageTopologyResolver` 生成页面拓扑上下文，例如单输入框、并列多输入框、矩阵型输入框、实验答题页、动态方案表。

`page_type` 是 L2 / Registry 中的页面拓扑类型，`algorithmId` 是后端 pipeline 配置和算法注册表中的可执行算法 ID。二者必须显式映射，后端运行时按 `page_id + page_type + configured algorithmId` 校验一致性：

| `page_type` | 默认 `algorithmId` |
|---|---|
| `A1_FLOW` | `A1_FLOW_PAGE` |
| `B1_TEXT_SINGLE` | `B1_TEXT_SINGLE` |
| `B2_TEXT_MULTI_PARALLEL` | `B2_TEXT_MULTI_PARALLEL` |
| `B3_TEXT_MATRIX_EVALUATION` | `B3_TEXT_MATRIX_EVALUATION` |
| `C1_INFO_SELECTION` | `C1_INFO_SELECTION` |
| `D1_SIMULATION_ONLY` | `D1_SIMULATION_ONLY` |
| `D2_SIMULATION_QUESTION` | `D2_SIMULATION_QUESTION` |
| `E1_CHART_PLAN_DECISION` | `E1_CHART_PLAN_DECISION` |

推荐接口：

```java
public interface FieldRegistry {
    FieldDefinition requireField(String pageId, String fieldId);
    List<FieldDefinition> requiredFields(String pageId);
    String registryVersion();
}

public class FieldDefinition {
    private String pageId;
    private String fieldId;
    private String fieldRole;
    private boolean required;
    private Integer order;
    private String groupId;
    private String methodId;
    private String evaluationSide;
}
```

实现边界：

- 后端只能从 `START_PAGE.value.metadata` 读取 registry version/hash，不得从 `START_PAGE` 或任意事件 `metadata` 中读取字段语义作为准确信源。
- `TraceEventNormalizer` 只处理新标准事件的排序、去噪、幂等合并和质量校验，不负责旧 Mark JSON 历史数据转换。
- Field Registry 不应 banana 专属化；标准 `field_id` 是同构结构槽位 ID，主题展示名只能进入 module/display config。
- Content Registry 只补齐会被打标算法解释的可阅读内容语义，不机械收集所有装饰文案和按钮文案。
- 会被本页规则消费的未知 `field_id` / `content_id` 必须输出 FATAL；明确 `tagging_consumed=false` 的非关键装饰性内容可降为 WARN。
- Registry / RuleConfig hash 统一按 canonical JSON 计算，并排除 `*_hash`、`*Hash`、`registry_hash`、`content_hash`、`contract_hash` 等自引用字段。

### 3.4 RuleConfig

所有阈值必须配置化：

| 配置项 | 建议初始值 | 用途 |
|---|---:|---|
| `textDebounceMs` | 2000 | 前端 `TEXT_CHANGE` 防抖参考值 |
| `textThrottleCharDelta` | 10 | 文本变化节流参考值 |
| `thinkGapMs` | 5000 | 长停顿生成 `GENERATE_THINK` |
| `fluidMinPositiveDelta` | 10 | 流畅生成最低字符增长 |
| `fluidMinDurationMs` | 5000 | 流畅生成最短输入片段时长 |
| `textBlurMinChar` | 1 | 失焦文本最小有效字符数 |
| `editDistanceThreshold` | 1 | 编辑距离阈值 |
| `focusDebounceMs` | 300 | 焦点抖动过滤 |
| `modalMisclickMaxMs` | 300 | 资料卡误开过滤 |
| `minMeaningfulDwellMs` | 800 | 有效查看资料最低停留 |
| `deepReadMs` | 8000 | 深度阅读初始阈值 |
| `chartHoverMinMs` | 500 | 有效图表悬停 |
| `expRunDebounceMs` | 1000 | 相同参数重复实验去重 |
| `pageHiddenAdjustEnabled` | true | 是否剔除页面隐藏时长 |
| `usePriorExp` | true | 第 9-11 页是否读取第 8 页前序实验 |

阈值说明：

- 这些值是 v2.1 初始工程阈值，不是理论常量或能力断言阈值。
- RuleConfig 必须声明 `ruleConfigVersion`、`timeUnit=ms`、阈值类型和非负范围。
- 上线后必须根据真实学生数据分布校准。
- 调整阈值不应改动规则代码，只改配置。

## 5. 全局规则

### 5.1 页面生命周期事件

生命周期事件只汇总到 `pageMetadata.lifecycle`，不生成 L3 标签。

生成条件：

- 页面存在 `START_PAGE` 事件时，记录 `lifecycle.startPage=true`、`page_index`、`page_type`、`flow_context`、`client_page_begin_time`。
- 出现 `SUBMIT_ATTEMPT.validation_status=success` 时，记录 `lifecycle.submitSuccess=true`、`submit_attempt_id`、`submit_trigger`、`required_fields_status`、`missing_fields`、`page_duration_ms`、`hidden_duration_total_ms`。
- 出现 `TASK_FINISH` 时，记录 `lifecycle.taskFinish=true` 和任务完成时间。

边界：

- `START_PAGE`、成功提交和 `TASK_FINISH` 只说明页面 / 任务生命周期，不代表阅读、理解、实验、反思或答案质量。
- 后端仍必须读取 `START_PAGE.value.metadata` 中的 registry version/hash；缺失时仍输出 `MISSING_START_PAGE` FATAL。
- 成功 `SUBMIT_ATTEMPT` 仍用于提交审计和页面时长计算，但不再生成 `SUBMIT_PAGE`。

### 5.2 `INVALID_SUBMIT`

生成条件：

- `SUBMIT_ATTEMPT.validation_status=blocked`。

Metadata：

- `submit_attempt_id`
- `missing_fields`
- `error_msg`
- `blocked_reason`
- `attempt_index`

边界：

- 只说明发生了无效提交尝试。
- 如果之后学生回到材料、文本框、实验区，应另行生成 `REVIEW_AFTER_BLOCKED` 或 `CHECK_REVIEW`，并标记 `after_invalid_submit=true`。

### 5.3 页面时长修正

计算：

```text
page_duration_ms =
  last_event_time - START_PAGE.time - sum(PAGE_HIDDEN 到 PAGE_VISIBLE 的隐藏时长)
```

要求：

- 若 `PAGE_HIDDEN` 无配对 `PAGE_VISIBLE`，记录数据质量告警。
- 隐藏时长不用于 `GENERATE_THINK`、阅读时长或页面停留解释。

## 6. 文本输入规则

适用页面：第 2、5、6、12 页。

其中第 2、5、6 页同属 `01轨迹上报规范.md` 中的 B 文本生成大类，后端应复用同一套文本规则；差异不靠重写文本规则解决，而靠 `fieldRegistryVersion + page_id + field_id` 查询静态 Field Registry 来解释字段拓扑：

| 页面 | 拓扑子类 | 后端解释重点 |
|---|---|---|
| 第 2 页 | `B1_TEXT_SINGLE` | 1 个 `input_question_1` 字段，文本修改即修改该页唯一输出 |
| 第 5 页 | `B2_TEXT_MULTI_PARALLEL` | 3 个并列 `input_idea_*` 字段，跨字段切换可作为跨方案查看 / 比较证据 |
| 第 6 页 | `B3_TEXT_MATRIX_EVALUATION` | 3 方法 x 2 维度字段，跨字段切换需结合 `method_slot` 与 `evaluation_side` 解释 |

实现要求：

- `TEXT_FOCUS`、`TEXT_CHANGE`、`TEXT_BLUR` 的基础规则不得按页面复制三份。
- Java 规则应先通过 `FieldRegistry` 建立 `field_id -> field_role/method_slot/evaluation_side/required/content_key` 的索引。
- 如果缺少 Field Registry 或 registry 版本不匹配，第 5、6 页可按约定字段 ID 临时兜底，但必须输出数据质量告警；正式环境应阻断上线。
- 后端不得信任前端运行时动态传来的字段语义覆盖静态 registry。

标准字段 ID 必须与 `01轨迹上报规范.md` 保持一致：

| 页面 | 标准字段 |
|---|---|
| Page02 | `input_question_1` |
| Page05 | `input_idea_1`、`input_idea_2`、`input_idea_3` |
| Page06 | `method_1_advantage`、`method_1_disadvantage`、`method_2_advantage`、`method_2_disadvantage`、`method_3_advantage`、`method_3_disadvantage` |
| Page12 | `chart_evidence_1`、`plan_table`、`reason_text`；动态方案行通过 `row_id` 关联，方案参数通过 `param_id=plan_param_1/plan_param_2/...` 关联 |

### 6.1 `GENERATE_THINK`

生成条件，满足任一：

- `TEXT_FOCUS` 到第一次有效 `TEXT_CHANGE` 的间隔 `>= thinkGapMs`。
- 同一 `field_id` 内相邻 `TEXT_CHANGE` 间隔 `>= thinkGapMs`，且不是 `PAGE_HIDDEN` 造成。

Metadata：

- `field_id`
- `focus_to_first_input_ms`
- `pause_ms`
- `pause_position`
- `evidence_event_ids`

边界：

- 只能说明学生在输入前或输入中出现等待 / 停顿。
- 不能直接证明学生在深度思考，也不能证明答案更好。

### 6.2 `GENERATE_FLUID`

生成条件，建议同时满足：

- 同一 `field_id` 内出现连续正向 `TEXT_CHANGE`。
- 累计 `positive_char_delta >= fluidMinPositiveDelta`。
- 输入片段持续时间 `>= fluidMinDurationMs`，或存在多次有效 `TEXT_CHANGE`。
- 最终 `TEXT_BLUR.value_after` 非空，字符数 `>= textBlurMinChar`。

Metadata：

- `field_id`
- `positive_char_delta`
- `change_event_count`
- `input_duration_ms`
- `final_char_count`

边界：

- 不使用逐字级按键间隔判断，因为前端已按防抖 / 节流合并。
- 只说明持续生成过程，不证明文本质量高。

### 6.3 `REVISE`

生成条件，满足任一：

- 同一 `field_id` 内出现负向 `char_delta`。
- `TEXT_BLUR.value_after` 与 `TEXT_FOCUS.value_before` 或前一快照编辑距离 `>= editDistanceThreshold`。
- 前后文本 hash 不同且差异不是纯追加。

Metadata：

- `field_id`
- `edit_distance`
- `char_delta_total`
- `revision_count`
- `revision_type`: `delete` / `replace` / `reorder` / `mixed`

边界：

- 不得用 `value_after < value_before` 这种字符串大小比较判断修正。
- `REVISE` 只说明发生修改，不说明修改质量。

### 6.4 `CROSS_FIELD_REVIEW`

适用页面：第 5、6、12 页。

生成条件：

- 已经在某个字段产生有效输入后，又聚焦另一个字段；
- 之后回到前一个字段，或在多个字段间来回切换；
- 切换不是 `focusDebounceMs` 内的焦点抖动。

Metadata：

- `field_switch_count`
- `visited_field_ids`
- `return_to_previous_field_count`
- `completed_field_count`

边界：

- 这是跨方案 / 跨评价维度 / 跨理由字段检查的弱证据。
- 不直接等同于高质量比较。
- `CROSS_BOX_REVIEW` 仅作为旧文档兼容别名，正式标签名统一为 `CROSS_FIELD_REVIEW`。

### 6.5 字段完成状态（L4 候选，本阶段不强制输出）

适用页面：所有含必填字段的页面。

本阶段用途：

- 后端必须能够读取 `SUBMIT_ATTEMPT.missing_fields` 并用于 `INVALID_SUBMIT` / `pageMetadata.lifecycle.submitSuccess` 规则判断。
- 后端可以在内部计算字段完成状态用于校验，但不要求输出独立 L4 `FIELD_COMPLETENESS` 对象。

后续 L4 生成条件：

- 根据 Field Registry 取得当前页 `required_fields`；
- 根据最近一次 `SUBMIT_ATTEMPT.missing_fields` 或字段最后状态计算完成情况；
- 所有必填字段非空时输出 `complete=true`，否则输出缺失字段列表。

Metadata：

- `required_field_ids`
- `missing_field_ids`
- `completed_field_count`
- `required_field_count`
- `useInHmm=false`

边界：

- 只判断字段是否完成，不判断答案是否正确，也不读取 `answerList`。
- `missing_field_ids` 必须使用标准 `field_id`、`question_id` 或 `row_id` 表达，不能使用页面文案。
- `FIELD_COMPLETENESS` 是页面状态 / L4 Metadata，本阶段不作为 L3 行为标签输出，也不进入 HMM 观测序列。

### 6.6 `REVIEW_AFTER_BLOCKED`

适用页面：所有会出现 `SUBMIT_ATTEMPT.validation_status=blocked` 的页面。

生成条件：

- 先出现一次被拦截的 `SUBMIT_ATTEMPT`；
- 之后学生发生有效回看或补改行为，例如 `TEXT_FOCUS`、`TEXT_CHANGE`、`OPEN_MODAL`、`CHART_HOVER`、`EXECUTE_EXP`、`SELECT_ANSWER`；
- 行为发生在下一次成功提交之前。

Metadata：

- `blocked_submit_attempt_id`
- `missing_fields`
- `review_event_type`
- `review_delay_ms`

边界：

- 该标签说明学生在失败提交后有补查 / 补改行为，不说明补改质量。
- 不能仅凭 `INVALID_SUBMIT` 自动生成，必须在无效提交后出现后续有效回看、修改或补全证据。

## 7. 材料查看与多选规则

### 7.1 `READ_CHAT` / `REVIEW_CHAT`

适用页面：第 2 页。

输入事件：

- `CHAT_SCROLL`
- `CONTENT_EXPOSE`
- `CONTENT_ACTIVATE`
- 可选增强事件 `CHAT_VIEWPORT_ENTER` / `CHAT_VIEWPORT_LEAVE`

`CHAT_SCROLL`、`CONTENT_EXPOSE`、`CONTENT_ACTIVATE` 是 L2 标准事件，不直接作为最终 L3 / HMM 标签。

`READ_CHAT` 生成条件：

- 首次 `TEXT_FOCUS` 或首次有效 `TEXT_CHANGE` 之前，出现用户主动触发的有效 `CHAT_SCROLL`、`CONTENT_ACTIVATE`，或 `expose_source=user_scroll` 的 `CONTENT_EXPOSE`。
- 如果只有聊天自动播放曝光，没有主动滚动 / 停留 / 触摸，则只进入 `reading_features.passive_auto_exposure`，默认不生成可进入 HMM 的 `READ_CHAT`。

`REVIEW_CHAT` 生成条件：

- 已发生 `TEXT_CHANGE` 或 `TEXT_BLUR` 后，再次出现有效 `CHAT_SCROLL`、`CONTENT_ACTIVATE` 或用户触发的 `CONTENT_EXPOSE`。
- 输入后 `CHAT_SCROLL.scroll_direction=up` 是较强的回看旧气泡证据。
- 同一 Page02 聊天 evidence 默认优先生成 `READ_CHAT` 或 `REVIEW_CHAT`；除非本规范另有明确要求，不再基于同一证据重复生成 `CHECK_REVIEW`。

Metadata：

- `label_source`: `active_scroll` / `content_hover` / `content_touch` / `user_exposure`
- `evidence_content_ids`
- `key_content_covered`
- `scroll_up_count`
- `before_first_input` / `after_input`
- `useInHmm`: 主动阅读 / 回看标签可为 `true`；仅自动曝光推断必须为 `false`

后续 L4 阅读特征候选：

- `chat_bubble_total_count`
- `chat_bubble_user_exposed_count`
- `chat_bubble_activated_count`
- `key_bubble_total_count`
- `key_bubble_user_exposed_count`
- `key_bubble_activated_count`
- `total_user_exposed_ms`
- `total_hover_ms`
- `first_user_chat_action_time`
- `chat_revisit_count_after_input`

本阶段只要求生成 `READ_CHAT` / `REVIEW_CHAT` / `REVIEW_AFTER_BLOCKED` 等 L3 标签，并在标签 metadata 中保留 `evidence_content_ids`、`key_content_covered`、`scroll_up_count` 等证据。上述阅读特征暂不作为独立 L4 输出。

边界：

- 阅读相关标签只能说明查看或回看聊天材料，不能直接证明阅读理解。
- `is_key_evidence=true` 的气泡被曝光或激活，只能说明关注过关键证据，不能说明理解了关键证据。

### 7.2 `SCAN_INFO` 与 `DEEP_READ`

适用页面：第 3 页。

生成条件：

- `OPEN_MODAL` 与同一 `card_id` 的 `CLOSE_MODAL` 成对。
- `dwell_ms < modalMisclickMaxMs`：视为误触，不生成标签。
- `modalMisclickMaxMs <= dwell_ms < deepReadMs`：生成 `SCAN_INFO`。
- `dwell_ms >= deepReadMs`：生成 `DEEP_READ`。

Metadata：

- `card_id`
- `card_type`
- `dwell_ms`
- `open_count`

边界：

- `DEEP_READ` 只说明停留较久，不证明理解。
- 同一资料卡查看 evidence 默认只生成一种阅读主标签：`SCAN_INFO` 或 `DEEP_READ`。阅读深浅进入标签 metadata，避免同时输出多个高度重叠阅读标签污染 HMM。

### 7.3 `SELECT_OPTION` / `REVISE_SELECTION`

适用页面：第 3 页。

生成条件：

- 前端上报 L2 `CHECKBOX_TOGGLE`。
- `CHECKBOX_TOGGLE.checked=true` 生成 L3 `SELECT_OPTION`。
- `CHECKBOX_TOGGLE.checked=false` 生成 L3 `REVISE_SELECTION`，或生成 `REVISE` 且必须带 `revision_target=selection`。

Metadata：

- `option_id`
- `checked`
- `selected_option_ids`
- `toggle_count_by_option`
- `revision_target=selection`（取消 / 改选时）

噪音处理：

- 同一 `option_id` 1 秒内连续 3 次及以上快速切换，应合并并标记 `rapid_toggle=true`。
- `CHECKBOX_TOGGLE` 保留为 L2 standard event，不作为 HMM 主观测标签。

### 7.4 第 3 页 `CHECK_REVIEW`

生成条件：

- 已至少勾选一个选项后，再打开资料卡；
- 或发生 `INVALID_SUBMIT` 后，再打开资料卡或调整选项。

Metadata：

- `review_phase`: `after_selection` / `after_invalid_submit`
- `reviewTargetType`: `material`
- `reviewed_card_ids`
- `evidence_event_ids`

## 8. 实验规则

适用页面：第 8、9、10、11 页。

### 8.1 `SET_EXP_PARAM`

生成条件：

- 出现 `SET_EXP_PARAM` 事件。
- 同一参数同一值的重复设置可合并。

边界：

- `SET_EXP_PARAM` 说明学生进行了参数探索 / 数据获取准备，不直接断言实验设计能力。

Metadata：

- `param_name`
- `value_before`
- `value_after`
- `param_snapshot`
- `param_change_count`

### 8.2 `EXECUTE_EXP`

生成条件：

- 出现 `EXECUTE_EXP` 事件。

去重规则：

- 同一页面、同一 `param_snapshot`、`expRunDebounceMs` 内重复执行，只保留第一次为有效标签。
- 被丢弃的重复事件不删除原始证据，但在 Metadata 标记 `duplicate_dropped=true`。
- 前端初始防双击窗口为 `1500ms`，窗口内重复点击应体现为 `click_debounce_applied=true`，后端不得将其计为新的有效实验执行。

Metadata：

- `exp_run_id`
- `param_snapshot`
- `result_snapshot`
- `click_debounce_applied`
- `run_index`
- `deduped_run_count`

边界：

- `result_snapshot` 是实验结果，不是学生行为。
- `EXECUTE_EXP` 只说明学生执行了实验，不说明实验设计合理。

### 8.3 `RESET_EXP`

生成条件：

- 出现 `RESET_EXP` 事件。

Metadata：

- `param_snapshot_before_reset`
- `reset_count`

### 8.4 第 8 页页面级过程标记

第 8 页没有答题区，不生成 `SELECT_ANSWER`。

页面级过程标记：

| 标记 | 条件 |
|---|---|

## 9. 第 12 页综合方案页规则

第 12 页主能力维度为“探究方案评估能力”，同时允许辅助证据：

生成条件：

### 9.5 文本内容边界：当前不输出 `ARGUMENTATION`

本阶段不把 `ARGUMENTATION` 作为正式 L3 / HMM 标签：

- 轨迹打标目标是行为过程标签，不是开放文本内容评价。
- 判断理由文本是否有数字、比较词、因果词属于文本内容特征提取，不是纯轨迹行为打标。
- 表层词线索不能证明学生完成有效论证，不能作为能力断言。
- `answerList` 仍是评分答案数据源；开放文本质量由评分 / 评价模块处理。
- 为保持数据最小化，`detail_json` 不保存理由原文或文本摘要。

如后续确需文本表层线索，可新增 `REASON_SURFACE_CUES`，允许临时读取文本、提取数字 / 比较 / 因果等表层 cue 后丢弃原文，并设置 `useInHmm=false`；真正的论证质量、理由质量、答案正确性由评分模块或后续 L4/L5 评价模块处理。

## 11. 数据质量校验

Java 算法必须输出 `dataQualityWarnings`，每项必须包含 `severity`：

| 告警 | 默认级别 | 条件 |
|---|---|---|
| `INVALID_TRACE_EVENT_SCHEMA` | FATAL | `operationList[*]` 不符合当前合同版本的 event schema；当前香蕉实现为 `event_schema.v2.2.json` |
| `MISSING_START_PAGE` | FATAL | 页面无 `START_PAGE`，无法读取合同版本 / hash |
| `EVENT_ORDER_UNRESOLVABLE` | FATAL | 事件无法按 `eventSeq` 与时间可靠排序 |
| `MISSING_FIELD_REGISTRY` | FATAL | 无法加载指定版本 Field Registry |
| `FIELD_REGISTRY_VERSION_MISMATCH` | FATAL | 前端上报的 registry 版本或 hash 与后端不一致 |
| `MISSING_CONTENT_REGISTRY` | FATAL | 无法加载当前页面规则消费的 Content Registry |
| `CONTENT_REGISTRY_VERSION_MISMATCH` | FATAL | Content Registry 版本或 hash 与后端不一致 |
| `UNKNOWN_FIELD_ID` | FATAL / WARN | 被规则消费时 FATAL；明确 `tagging_consumed=false` 的非关键字段可 WARN |
| `UNKNOWN_CONTENT_ID` | FATAL / WARN | 被规则消费时 FATAL；明确 `tagging_consumed=false` 的装饰性内容可 WARN |
| `DUPLICATE_EVENT_SEQ` | WARN | 重复事件序号可由工程噪声过滤处理 |
| `TIME_ORDER_ADJUSTED` | WARN | 轻微时间倒序但可按 `eventSeq` 修正 |
| `TIME_PRECISION_TOO_LOW` | WARN | 事件时间只有秒级或无法确认毫秒精度 |
| `TIME_PARSE_FAILED` | FATAL / WARN | 关键时间字段无法解析且影响排序为 FATAL，否则 WARN |
| `MISSING_TARGET_ID` | WARN | 关键事件无 `target_id` 但不影响规则消费 |
| `MISSING_FIELD_ID` | FATAL | 文本、题目、动态表格等关键事件缺少标准 `field_id` |
| `MISSING_CONTENT_ID` | FATAL | 阅读类规则消费的事件缺少标准 `content_id` |
| `TEXT_BLUR_WITHOUT_FOCUS` | WARN | 出现 `TEXT_BLUR` 但无前置 `TEXT_FOCUS` |
| `TEXT_CHANGE_RAW_TEXT_PRESENT` | WARN | `TEXT_CHANGE.value_after` 出现过程明文 |
| `ANSWER_LIST_USED_BY_TAGGER` | FATAL | 轨迹打标算法读取或分析了 `answerList`，违反答案评分与轨迹打标边界 |
| `EXP_RESULT_AS_STUDENT_EVENT` | WARN | 实验结果被上报为独立学生行为 |
| `PAGE08_HAS_SELECT_ANSWER` | FATAL | 第 8 页出现答题事件 |
| `QUESTION_INDEX_MISMATCH` | FATAL | 第 9-11 页题号与页面不匹配 |
| `ROW_ID_MISSING` | FATAL | 第 12 页动态表格事件缺 `row_id` |

处理规则：

- `severity=FATAL` 时，执行项标记 `FAILED`，不写正常 `sequence_json`；可以写失败诊断 `detail_json`，但 `executionStatus=FAILED` 且 `labels` 为空或不生成。
- `severity=WARN` 时，执行项仍可 `SUCCESS`，正常写 `sequence_json + detail_json`。
- `suppressedEvents` 记录事件级过滤 / 合并明细；如果 suppression 影响页面整体质量，再额外生成 WARN。
- 所有被过滤或合并的事件必须进入 `suppressedEvents` 或 `dataQualityWarnings`，不能静默删除。

## 13. 证据与推理依据

### 13.1 为什么能力维度按页面主任务归属

如果每个微操作都单独映射能力维度，会导致能力维度碎片化。例如第 5 页核心任务是提出三个方案，如果把 `TEXT_FOCUS` 映射成科学问题能力、`TEXT_CHANGE` 映射成论证能力、`SUBMIT_PAGE` 映射成决策能力，后续 HMM / Viterbi 路径将难以解释学生到底在什么主任务上表现出何种过程特征。

因此本规范采用：

- 单一任务页：一页一个主能力维度。
- 综合任务页：页面主维度 + 操作级辅助维度。
- 流程页：非能力标签。

### 13.2 为什么 L3 标签不直接判断能力水平

ECD 强调“证据规则”服务于对能力主张的推断，但一个轨迹事件通常只是局部证据。`EXECUTE_EXP` 说明学生执行了实验，不说明实验设计合理；`WRITE_REASON` 说明学生输入了理由，不说明理由质量、论证质量或答案正确。因此能力水平应在 L5+ 模型推断或文本评分中完成。

### 13.3 为什么需要时序标签

学习分析中的时间分析强调事件顺序、持续时间和状态转移。第 9-11 页中“先实验后答题”和“先答题后回实验核对”是不同策略；如果只保存最终答案，将无法区分。HMM 也需要离散、稳定、可解释的观测标签序列，因此后端必须保留标签顺序和证据事件链。


### 13.3.1 为什么页面生命周期不进入 HMM

HMM 的观测符号应表示学生可观察的探究行为，并可被合理解释为隐藏认知状态的外显证据。`START_PAGE`、成功 `SUBMIT_ATTEMPT` 和 `TASK_FINISH` 是页面或任务的物理生命周期边界，几乎所有页面都会出现，不能对应阅读理解、实验探索或反思修正等隐藏状态。

如果把这些生命周期边界放入 `sequence_json`，转移矩阵会大量学习 `START_PAGE -> 行为`、`行为 -> SUBMIT_PAGE` 这类页面结构转移，稀释对核心探究路径的敏感度；序列起点本应由 HMM 初始状态分布处理，页面结束应由序列终止或 `pageMetadata.lifecycle` 表达。因此生命周期事件必须保留为审计和上下文证据，但不得作为 L3 / HMM 观测符号。

### 13.4 为什么文本输入不能逐字打标

前端已按防抖 / 节流合并 `TEXT_CHANGE`，后端不能假设每个字符都有事件。`GENERATE_THINK`、`GENERATE_FLUID`、`REVISE` 应基于聚焦时间、合并后的文本变化、字符数、停顿、失焦快照综合判断，而不是逐字键盘间隔。

### 13.5 为什么字段语义必须来自静态 Field Registry

字段语义不是学生行为，而是前后端共同遵守的测评契约。第 2 页、5 页、6 页都可能产生 `TEXT_FOCUS` / `TEXT_CHANGE` / `TEXT_BLUR`，但 `input_question_1`、`input_idea_1`、`method_1_advantage` 代表的字段角色、必填规则和跨字段关系不同。如果让前端在事件流中动态上报字段语义，会把稳定契约变成运行时数据，增加版本不一致、重复上报和后端信任边界问题。

因此本规范要求：

- 前端事件只引用 `field_id`，不动态声明字段语义。
- 后端按 `fieldRegistryVersion + page_id + field_id` 查询 Field Registry。
- `field_id` 的具体命名由本标准定义为跨子模块复用的结构槽位 ID；前端只负责把真实控件绑定到标准 ID，不能自行改名。
- `TraceEventNormalizer` 只处理新标准事件，不负责把旧 Mark JSON 历史数据映射为标准事件。
- 若 registry 缺失、版本不一致或字段未知，应输出 `MISSING_FIELD_REGISTRY`、`FIELD_REGISTRY_VERSION_MISMATCH`、`UNKNOWN_FIELD_ID` 等数据质量告警。

### 13.6 为什么字段 ID 不能绑定具体内容主题

香蕉变黑、蜂蜜黏度、蒸馒头等子模块如果页面拓扑一致，学生行为证据的结构也一致：第 6 页都是 3 种方法 x 优缺点评价，第 12 页都是证据图表 + 方案表 + 理由说明。若把字段命名为 `method_image_advantage` 或 `chart_black_ratio`，后端规则就会绑定香蕉变黑内容，蜂蜜或馒头模块必须复制一套算法。

因此本规范采用结构槽位命名：

- 第 6 页使用 `method_1_advantage`、`method_2_disadvantage` 等结构字段，Field Registry 再说明 `method_1` 在香蕉实例中显示为“图像法”，在其他模块中显示为对应方法。
- 第 12 页使用 `chart_evidence_1`、`plan_table`、`reason_text`，Field Registry 再说明图表指标、横纵轴含义和方案参数含义。
- Java 规则只读取 `method_slot`、`evaluation_side`、`param_id` 和字段完成状态；具体主题词只进入展示、评分和报告解释层。

### 13.7 为什么聊天气泡要全量注册但分层打标

第 2 页聊天区以动画逐条呈现，新气泡会把旧气泡顶出手机框。自动呈现只是系统提供阅读机会，不能直接证明学生阅读；但学生主动滚动回看、鼠标停留或触摸气泡，是比自动曝光更强的阅读注意证据。

因此本规范要求：

- 所有聊天气泡进入 Content Registry，不只注册关键气泡；否则无法分析阅读覆盖率、跳读路径和输入后回看。
- 前端只把达到阈值的 `CONTENT_EXPOSE`、`CONTENT_ACTIVATE`、`CHAT_SCROLL` 写入 `operationList`，不逐帧上报鼠标移动和滚动。
- 后端在本阶段把这些 L2 事件聚合为 `READ_CHAT`、`REVIEW_CHAT`、`REVIEW_AFTER_BLOCKED` 等 L3 标签；`reading_features` 属于后续 L4 扩展，不在本阶段强制输出。
- 阅读标签只能说明查看或回看材料，不能证明阅读理解；关键气泡覆盖率只能作为后续模型或人工分析的弱证据。

### 13.8 为什么不做旧 Mark JSON 历史迁移

现有前端标准使用 `pageNumber`、`pageDesc`、`operationList`、`answerList` 的 Mark JSON 结构，单条操作只有 `code`、`targetElement`、`eventType`、`value`、`time`。这些数据的时间精度、字段语义和事件粒度与新标准差距较大，强行转换会产生看似精确但实际不可靠的行为证据。

因此本规范不要求旧数据迁移：

- `香蕉变黑轨迹数据现有标准.md` 只作为现状差距分析依据。
- 前端子模块重构后，新产生的轨迹数据仍使用现有页面批量 Mark JSON envelope 上报，但 `operationList` 内部必须按 `01轨迹上报规范.md` 的标准事件结构存储。
- `TraceEventNormalizer` 负责新标准事件排序、去噪、毫秒时间校验、Field Registry 校验和重复事件合并，不负责历史转换。
- 新标准必须要求毫秒级时间，否则 `GENERATE_THINK`、`GENERATE_FLUID`、`CROSS_FIELD_REVIEW`、实验重复点击去重等规则会失真。
- 如果后续为了研究目的临时读取旧 Mark JSON，必须明确标记为低精度现状分析，不能混入正式新标准打标结果。

### 13.9 为什么保留 `answerList` 但不用于轨迹打标

评分与轨迹打标服务回答的是两个不同问题。评分需要知道学生最后提交了什么答案；轨迹打标需要知道学生如何阅读、输入、修改、实验、回看和提交。

因此本规范要求：

- `answerList` 必须保留，并作为评分 / 答案判定的权威数据源。
- 轨迹打标只消费标准事件序列和 Field Registry，不读取 `answerList`。
- 文本事件中的 `value_after` 是过程快照，只用于行为证据，不作为评分答案。
- 如果 Java 打标实现为了生成轨迹标签读取 `answerList`，必须输出 `ANSWER_LIST_USED_BY_TAGGER` 并修正实现边界。
