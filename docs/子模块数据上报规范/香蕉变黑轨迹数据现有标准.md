## 一、数据提交总体架构

```
用户交互 → context.logOperation() → operationLogs[]
         → context.collectAnswer() → answerDraft.pageAnswers[]
                              ↓
         点击"下一步" → validatePage() → defaultSubmit()
                              ↓
         AssessmentPageFrame → usePageSubmission → buildMark()
                              ↓
         POST /stu/saveHcMark (FormData: { batchCode, examNo, mark: JSON.stringify(markObj) })
```

每次提交的数据是一条页面级别的 `mark` 记录，包含该页面上所有操作轨迹和最终答案。

## 二、提交数据结构（Mark JSON）

```json
{
  "pageNumber": "1.03",
  "pageDesc": "香蕉的奥秘",
  "operationList": [ /* 操作轨迹列表，详见第三节 */ ],
  "answerList": [ /* 答案列表，详见第六节 */ ],
  "beginTime": "2026-06-03 10:30:00",
  "endTime": "2026-06-03 10:30:20",
  "imgList": []
}
```

字段说明：

| 字段 | 格式 | 说明 |
|---|---|---|
| `pageNumber` | `"<stepIndex+1>.<subPageNum补零2位>"` | 复合页号，如 stepIndex=0, subPageNum=3 → `"1.03"` |
| `pageDesc` | 字符串 | 页面描述，带 Flow 前缀 |
| `operationList` | 数组 | 操作轨迹列表，`code` 从 1 递增 |
| `answerList` | 数组 | 答案列表，`code` 从 1 递增 |
| `beginTime` | `"YYYY-MM-DD HH:mm:ss"` | 页面进入时间 |
| `endTime` | `"YYYY-MM-DD HH:mm:ss"` | 页面提交时间 |
| `imgList` | 数组 | 图片列表（当前为空） |

外部包裹为 FormData：

```
batchCode: string
examNo: string
mark: JSON.stringify(上述 mark 对象)
```

## 三、操作轨迹（operationList）记录规范

### 3.1 单条操作记录结构

```json
{
  "code": 1,
  "targetElement": "P1.03_科学问题输入",
  "eventType": "input_change",
  "value": "{\"prev\":\"\",\"next\":\"为什么\"}",
  "time": "2026-06-03 10:30:10"
}
```

| 字段 | 说明 |
|---|---|
| `code` | 递增序号，从 1 开始，由 `OperationSequence` 自动分配 |
| `targetElement` | 目标元素标识，自动添加 `P{pageNumber}_` 前缀 |
| `eventType` | 事件类型，取值见 3.2 节 |
| `value` | 事件值，字符串类型；对象会被 `JSON.stringify` 序列化 |
| `time` | 时间戳，格式 `YYYY-MM-DD HH:mm:ss` |

### 3.2 targetElement 命名规范

**格式：** `P{pageNumber}_{元素名称}`

**自动前缀规则：**

- 前缀由 `buildTargetElementPrefix(pageNumber)` 生成，如 `P1.03_`
- 页面组件只需传元素名称（如 `科学问题输入`），logger 自动拼接
- 前缀正则校验：`/^P[1-9]\d*\.\d{2}_/`

**保留元素（不加前缀）：**

`page`、`页面`、`next_button`、`下一步按钮`、`下一页按钮`、`prev_button`、`上一步按钮`、`module`、`flow_context`、`task_timer`、`countdown`

### 3.3 事件类型（eventType）完整列表

香蕉变黑子模块使用的 22 种事件类型：

| 事件类型 | 使用页面 | 记录内容 |
|---|---|---|
| `page_enter` | 所有 14 个页面 | 页面加载完成 |
| `flow_context` | 所有页面（自动注入） | Flow 编排元数据 |
| `page_exit` | AssessmentPageFrame | 页面离开 |
| `next_click` | 下一页按钮 | 提交按钮点击 |
| `click` | Page03(对话)、Page13(表格操作) | 通用点击 |
| `click_blocked` | Page01(倒计时未完)、验证失败 | 被阻止的点击 |
| `change` | Page13(下拉选择) | 下拉框值变化 |
| `input_focus` | Page03/06/07(文本框) | 文本框获得焦点 |
| `input_change` | Page03/06/07/13(文本框) | 文本框内容变化 |
| `input_blur` | Page03/06/07(文本框) | 文本框失去焦点 |
| `radio_select` | Page10/11/12(选择题) | 单选项被选中 |
| `checkbox_check` | Page01 | 复选框勾选 |
| `checkbox_uncheck` | Page01 | 复选框取消 |
| `timer_complete` | Page01 | 倒计时结束 |
| `reading_complete` | Page04 | 阅读材料完成 |
| `simulation_timing_started` | SimulationPanel | 模拟实验开始 |
| `simulation_run_result` | SimulationPanel | 模拟实验结果 |
| `simulation_operation` | SimulationPanel | 天数增减/重置 |
| `page_submit_success` | usePageSubmission | 提交成功 |
| `page_submit_failed` | usePageSubmission | 提交失败 |
| `auto_submit` | usePageSubmission | 超时自动提交 |
| `timeout` | usePageSubmission | 会话超时 |

> 全局共定义 44 种事件类型（含 G4 拖拽/键盘/对话等），完整列表见 `src/shared/services/submission/eventTypes.js`。

### 3.4 每页固定事件序列

每个页面的 `operationList` 必须包含：

1. `page_enter`（页面加载时自动记录）
2. `flow_context`（第一个 `page_enter` 后自动注入）
3. 中间的交互事件（因页面而异）
4. `next_click` 或 `auto_submit`（至少一个）
5. `page_exit`

**flow_context 注入值：**

```json
{
  "flowId": "g8-banana-browning-experiment",
  "submoduleId": "g8-banana-browning-experiment",
  "stepIndex": 2,
  "moduleName": "8年级香蕉变黑科学探究",
  "pageId": "banana_mystery"
}
```

仅在每页首次 `page_enter` 时注入一次，页面切换（`clearOperations`）后重置。

## 四、文字输入框的记录方式

### 4.1 三阶段记录模式

文字输入使用 **focus → change(每次按键) → blur** 三阶段记录：

#### 阶段一：聚焦（input_focus）

```json
{
  "code": 3,
  "targetElement": "P1.03_科学问题输入",
  "eventType": "input_focus",
  "value": "聚焦",
  "time": "2026-06-03 10:30:05"
}
```

- `value` 固定为 `"聚焦"`（标记性字符串）
- 在对话类页面中可能为结构化信息：`"focus|role=student|idx=2|text=前50字符..."`

#### 阶段二：输入过程（input_change）— 每次按键触发

```json
{
  "code": 4,
  "targetElement": "P1.03_科学问题输入",
  "eventType": "input_change",
  "value": "{\"prev\":\"为什么\",\"next\":\"为什么香蕉\"}",
  "time": "2026-06-03 10:30:10"
}
```

- `value` 是 `JSON.stringify({ prev: "之前文本", next: "之后文本" })`
- **每次按键变化都触发**，完整记录输入过程
- 通过 `prev`/`next` 对比可精确还原每一次编辑操作

#### 阶段三：失焦（input_blur）

```json
{
  "code": 5,
  "targetElement": "P1.03_科学问题输入",
  "eventType": "input_blur",
  "value": "为什么香蕉会变黑",
  "time": "2026-06-03 10:30:15"
}
```

- `value` 是**失焦时的完整文本内容**（原始字符串）
- 代表该文本框在本次聚焦周期内的最终内容

### 4.2 删除事件（input_delete）— TrackedTextarea 专有

使用共享 `TrackedTextarea` 组件时，当新文本长度短于旧文本会额外记录：

```json
{
  "code": 4,
  "targetElement": "P1.06_想法1",
  "eventType": "input_delete",
  "value": "{\"action\":\"delete\",\"prevLength\":10,\"nextLength\":8}",
  "time": "2026-06-03 10:30:10"
}
```

- 在 `input_change` 之前插入
- 只记录长度变化，不记录具体删除了哪些字符

### 4.3 各页面文字输入框记录详情

| 页面 | subPageNum | 文本框元素 | targetElement 示例 | 记录事件 |
|---|---|---|---|---|
| Page03 香蕉的奥秘 | 3 | 科学问题单行输入 | `P1.03_科学问题输入` | focus → change(每次) → blur |
| Page06 方案设计 | 6 | 3 个 textarea（想法1/2/3） | `P1.06_想法1` 等 | focus → change → blur |
| Page07 方案评估 | 7 | 6 个 textarea（优缺点） | `P1.07_图像法优点` 等 | focus → change → blur |
| Page13 方案选择 | 13 | 理由 textarea | `P1.13_理由输入` | change(每次按键) |

### 4.4 输入轨迹的完整示例

假设用户在 Page03 依次输入"为"→"为什么"→"为什么香蕉会变黑"：

```
code=3  input_focus    value="聚焦"
code=4  input_change   value={"prev":"","next":"为"}
code=5  input_change   value={"prev":"为","next":"为什么"}
code=6  input_change   value={"prev":"为什么","next":"为什么香蕉"}
code=7  input_change   value={"prev":"为什么香蕉","next":"为什么香蕉会变黑"}
code=8  input_blur     value="为什么香蕉会变黑"
```

## 六、答案列表（answerList）记录规范

```json
{
  "answerList": [
    {
      "code": 1,
      "targetElement": "根据左侧对话，请写出接下来小明要探究的科学问题？",
      "value": "为什么香蕉会变黑"
    }
  ]
}
```

| 字段 | 说明 |
|---|---|
| `code` | 来自 `QUESTION_CODE_MAP` 的数字编号（1-17） |
| `targetElement` | 来自 `QUESTION_TEXT_MAP` 的**描述性中文题目文本** |
| `value` | 文本题：原始字符串；选择题：`"A. 3天"` 格式 |

### 6.1 题目编码映射

| code | 题目 ID | targetElement（题目描述） |
|---|---|---|
| 1 | Q1 | 根据左侧对话，请写出接下来小明要探究的科学问题？ |
| 2 | Q2 | 你认为影响香蕉变黑速度的因素有哪些？ |
| 3 | Q3a | 请写出你的第一个想法 |
| 4 | Q3b | 请写出你的第二个想法 |
| 5 | Q3c | 请写出你的第三个想法 |
| 6-11 | Q4a-Q4f | 图像法优点/缺点、观察法优点/缺点、称重法优点/缺点 |
| 12 | Q5 | 海南香蕉在常温下几天会变黑？ |
| 13 | Q6 | 常温储存下哪个品种香蕉最容易变黑？ |
| 14 | Q7 | 哪种温度条件下香蕉变黑速度最平缓？ |
| 15 | Q8_方案表格 | 实验方案表格数据（JSON 字符串） |
| 16 | Q8_最优方案 | 最优方案编号 |
| 17 | Q8_理由 | 选择理由文本 |

### 6.2 答案记录时机

- 通过 `collectAnswer({ targetElement, value })` **实时记录**
- 每次输入变化都更新 `answerDraft` 中的最新值
- 提交时由共享 `collectAnswers()` 适配器统一从 `answerDraft` 提取最终值

## 七、共享 Tracked 组件

项目在 `src/shared/components/tracking/` 下提供标准跟踪组件：

| 组件 | 自动记录的事件 |
|---|---|
| `TrackedInput` | `input_focus` → `input_change`(每次) → `input_blur` |
| `TrackedTextarea` | `input_focus` → `input_delete`(如有删除) → `input_change`(每次) → `input_blur` |
| `TrackedSelect` | `select_change`（单选为字符串，多选为 `{ selected: string[] }`） |
| `TrackedButton` | `click` / `next_click`（有 `requiredFields` 时可能触发 `click_blocked`） |

> 香蕉变黑子模块当前在各页面组件中**手动调用** `logOperation()`，未使用共享 Tracked 组件。两种方式产生的数据格式一致。

## 八、验证规范

### 8.1 Schema 层（Zod，`schema.ts`）

| 校验项 | 规则 |
|---|---|
| `pageNumber` | 必须匹配 `/^[1-9]\d*\.\d{2}$/` |
| `targetElement` 前缀 | 非保留元素必须匹配 `/^P\d+\.\d{2}_/` |
| `code` 递增 | `operationList[0].code === 1`，依次递增 |
| 必选事件 | 必须包含 `page_enter`、`page_exit`，以及 `next_click` 或 `auto_submit` 之一 |
| 输入事件 value | `input_focus`/`input_blur`：字符串；`input_change`：`{ prev, next }` JSON 或纯字符串；`input_delete`：`{ action, prevLength, nextLength }` |
| `flow_context` value | 必须是 `Record<string, unknown>` 或可解析的 JSON 字符串 |

### 8.2 validateMark 层（`validateMark.ts`）

| 校验项 | 规则 |
|---|---|
| 前缀匹配 | 非保留元素的 `targetElement` 前缀必须与当前页 `pageNumber` 一致 |
| 时间格式 | `YYYY-MM-DD HH:mm:ss` 或 ISO 8601 |
| 时间顺序 | `beginTime < endTime` |
| `code` 起始 | 从 1 开始 |
