# 01 轨迹上报规范

适用对象：香蕉变黑及同构科学探究交互任务的 13 个页面轨迹数据采集与上报。同构任务包括页面结构一致、仅内容主题不同的子模块，例如蜂蜜黏度、蒸馒头等；`Page01.PNG` ~ `Page13.PNG` 是香蕉变黑实例证据，不是字段命名必须绑定的内容边界。

本文档用于约束前端上报什么、不上报什么、如何上报，以及为什么这样上报。后端 `02打标规范.md` 必须以本文档定义的事件与字段为唯一输入口径。

版本状态（2026-06-06）：本文正文保留 v2.1 系列规范和示例；当前香蕉变黑前端运行时已经升级到 v2.2，实施时以 `engineering_contracts/event_schema.v2.2.json`、`field_registry.v2.2.json`、`content_registry.banana.v2.2.json`、`rule_config.v2.2.json` 和 `docs/superpowers/specs/2026-06-06-banana-trace-v2-2-frontend-implementation-design.md` 为准。v2.1 文件作为历史兼容补丁镜像保留，其中 `event_schema.v2.1.json` 已回补 `PAGE_IDLE` 以保证同版本 docs/runtime 副本一致。

## 1. 总体结论

同构科学探究任务的 13 个交互页面按 **操作行为大类 + 控件拓扑子类** 组织。前端采集的第一分类依据是学生可观察操作和控件拓扑，而不是具体主题内容或高阶能力维度；能力维度解释放在 `02打标规范.md` 和后续分析层完成。

| 操作大类 | 拓扑子类 | 对应页面 | 核心采集目的 |
|---|---|---|---|
| A 流程页 | A1 流程 / 过渡 / 结束 | 第 1、4、7、13 页 | 记录到达、停留、阶段流转或任务完成 |
| B 文本生成页 | B1 单输入框文本生成 | 第 2 页 | 记录材料查看、构思、文本生成、修正、提交状态 |
| B 文本生成页 | B2 多并列输入框文本生成 | 第 5 页 | 共用文本事件，额外记录 3 个方案字段的完成与跨框切换 |
| B 文本生成页 | B3 矩阵型多输入框评价 | 第 6 页 | 共用文本事件，额外记录 3 方法 x 2 维度字段的完成与跨维度切换 |
| C 资料判断页 | C1 资料卡片 + 多选判断 | 第 3 页 | 记录资料卡查看、资料停留、多选勾选 / 取消、回看检查、提交状态 |
| D 实验页 | D1 纯模拟实验 | 第 8 页 | 记录实验参数、执行实验、重置实验、是否跳过实验 |
| D 实验页 | D2 模拟实验 + 单选题 | 第 9、10、11 页 | 隔离记录实验区操作与当前页单选题作答 |
| E 综合决策页 | E1 图表 + 方案表 + 理由 | 第 12 页 | 记录图表查看、方案行操作、最佳方案选择、理由输入、提交状态 |

本阶段实施边界：

- 前端负责完成 L0 原始可观察动作的内部采集、L1 Registry ID 绑定，以及 L2 标准事件写入 `operationList`。
- 前端不得生成 L3 行为标签，不得生成 L4 过程特征，不得上报能力判断或策略结论。
- L4 特征聚合暂不作为本阶段交付内容；前端只需保证 L2 事件中的 `field_id`、`content_id`、时间、顺序和 metadata 足以支持后续 L3 / L4 扩展。

### 1.1 为什么采用“大类 + 子类”

本任务的采集标准需要同时满足两个目标：

- **前端记录事实**：同一类可观察操作应共用同一套事件定义。例如第 2、5、6 页都使用 `TEXT_FOCUS`、`TEXT_CHANGE`、`TEXT_BLUR`、`SUBMIT_ATTEMPT`，防抖 / 节流和文本快照规则完全一致。
- **保留控件拓扑契约**：同一事件在不同页面中的字段结构不同。第 2 页是 1 个输入框，第 5 页是 3 个并列方案输入框，第 6 页是 3 方法 x 2 维度的评价矩阵；它们可以共用文本事件，但不能共用同一套 `field_id` 集合、必填规则和跨字段关系。

因此，本文档不再把第 2、5、6 页拆成三个互不相关的“能力模板”，而是把它们归入 **B 文本生成页**，再用 B1/B2/B3 三个拓扑子类声明字段结构。这样既支持前端复用 `TextEventCollector`，也避免把字段拓扑差异隐藏到后端 `if-else` 或大量 nullable metadata 中。

### 1.2 同构任务复用原则

本标准的核心复用单位是“页面拓扑”，不是“香蕉变黑”这一具体内容主题。

- `field_id` 必须描述控件结构槽位，例如第 6 页第 1 种方法的优点，而不能绑定具体主题内容，例如“图像法优点”。
- 主题内容进入 Field Registry 的模块配置，例如 `display_name`、`content_key`、`option_label`、`measure_label`，不进入通用打标算法的字段命名。
- 后端 Java 打标算法只能依赖 `page_type`、`field_role`、`method_slot`、`evaluation_side`、`param_id`、`required` 等结构语义；不能写死香蕉、蜂蜜、馒头等主题词。
- 同一结构的子模块只需要替换 Field Registry 的内容配置和评分 `answerList`，不应复制一套新的轨迹打标算法。

### 1.3 理论依据

- **证据中心设计 ECD**：测评设计应先明确能力主张，再设计可观察行为证据，再定义证据规则。轨迹采集不能只记录“所有点击”，而要记录能支持后续推断的证据。参考 Mislevy、Almond、Lukas 的 ECD 报告，以及 Bennett 等关于复杂表现任务数据特征提取的报告。
- **学习分析与 EDM**：过程数据需要按任务类型组织成稳定特征，再进入模型或规则系统。不同任务类型暴露的认知过程不同，阅读、写作、实验、决策不能混用同一套特征。
- **时间序列学习分析**：后续 HMM / Viterbi / 过程挖掘依赖有序、可解释的事件序列；因此必须保留 `event_seq`、时间戳、页面上下文和控件上下文。
- **Web 事件工程依据**：浏览器 `input`、`change`、`focus`、`blur` 等事件语义不同，文本输入类轨迹必须区分“获得焦点”“输入变化”“失焦快照”，不能把它们混成一次提交值。

参考来源见文末“外部依据”。

## 2. 本地证据与推理过程

### 2.1 本地证据

本规范基于以下本地材料：

| 证据 | 用途 |
|---|---|
| `香蕉变黑两套标准文档（终稿）.pdf` | 已完成的逐页轨迹标准与打标标准，是本次重构的主证据 |
| `字段解释.md` | 解释 `TEXT_CHANGE`、`TEXT_BLUR`、`metadata`、防抖 / 节流等字段口径 |
| `第 8-11 页实验类页面统一注意事项.md` | 明确第 8 页与第 9-11 页实验类页面的边界 |
| `香蕉变黑轨迹数据现有标准.md` | 现有前端上报格式与事件类型，是现状差距分析依据 |
| `Page01.PNG` ~ `Page13.PNG` | 页面形式与控件结构证据 |

### 2.2 关键证据摘取

- 主文档显示第 1 页是背景页，第 2 页是单输入框页面，第 3 页是多选 / 资料卡页面，第 4 页是过渡页，第 5 页是三个输入框页面，第 6 页是六个评价输入框页面，第 8-11 页是模拟实验相关页面，第 12 页是方案选择与理由说明页。
- 辅助说明明确：第 8 页只记录实验操作，不记录答题行为；第 9-11 页既记录实验操作，也记录当前页单选题。第 9 页为 1 题 5 选项，第 10 页为 1 题 2 选项，第 11 页为 1 题 3 选项。
- 字段说明明确：文本输入不能逐字实时上报，应通过防抖或节流合并为 `TEXT_CHANGE`；最终完整文本由 `TEXT_BLUR` 上报。
- 现有标准中 `operationList` 字段包括 `code`、`targetElement`、`eventType`、`value`、`time`。其中 `targetElement` 使用 `P{pageNumber}_{元素名称}`，`eventType` 使用 `input_focus`、`input_change`、`input_blur`、`simulation_operation` 等旧事件名。
- 现有文本输入是 `input_focus → input_change(每次按键) → input_blur`，且 `input_change.value` 可能保存 `{ prev, next }` 过程明文。新标准需要把逐字输入合并为防抖 / 节流后的 `TEXT_CHANGE`，并把过程特征拆到 `metadata`。
- 现有前端已有 `mapping.ts`、`QUESTION_CODE_MAP`、`QUESTION_TEXT_MAP` 等静态配置基础。因此字段语义更适合作为前后端共享的静态 Field Registry，而不是由前端每页运行时上报。

### 2.3 推理链

1. 后端打标需要的是“可解释的行为证据”，不是原始浏览器噪音。
2. 每类页面能够产生的有效证据由可观察操作和控件结构共同决定：文本框产生输入过程证据，资料卡产生查看证据，实验控件产生参数与执行证据，单选题产生选择与改选证据。
3. 因此先按操作行为大类复用事件定义，再按控件拓扑子类声明字段集合、必填集合和跨字段关系。
4. 对高频低语义事件进行合并或丢弃：逐字输入、鼠标移动、无业务空白点击、系统自动渲染结果都不作为独立学生行为事件。
5. 对后续打标必须使用的上下文保留结构化字段：页面 ID、控件 ID、字段 ID、题号、参数快照、结果快照、校验状态、事件顺序。
6. 字段拓扑语义属于前后端契约，不属于学生行为轨迹；因此它应进入版本化 Field Registry，而不是混入 `operationList`、单条事件或 `START_PAGE.value.metadata` 的字段清单。
7. 旧数据不做历史迁移；现有 Mark JSON 只用于说明当前前端实现与新标准的差距。前端重构后必须直接按新标准上报 `field_id` 与防抖 / 节流后的标准事件。

## 3. 标准上报对象

当前前端采用页面批量上报：学生点击“下一页”后，完成当前页整体数据上报。新规范不改变现有页面级 Mark JSON 总体结构，只规范 `operationList` 中的轨迹事件结构；`answerList` 继续作为评分答案数据源独立保留。

### 3.1 页面级 Payload

```json
{
  "pageNumber": "1.03",
  "pageDesc": "香蕉的奥秘",
  "operationList": [
    {
      "code": 1,
      "targetElement": "P1.03_page",
      "eventType": "START_PAGE",
      "value": {
        "trace_id": "uuid",
        "page_id": "page_02_question_generation",
        "page_type": "B1_TEXT_SINGLE",
        "metadata": {
          "schema_version": "science-inquiry-trace-v2.1",
          "field_registry_version": "science-inquiry-field-registry-v2.1",
          "field_registry_hash": "optional-sha256",
          "content_registry_version": "science-inquiry-content-registry-banana-v2.1",
          "content_registry_hash": "optional-sha256",
          "task_family": "science_inquiry_13page",
          "module_id": "banana_browning",
          "initial_visible_content_ids": [
            "instruction_text_02_01"
          ],
          "main_instruction_visible": true,
          "viewport_state": "default_full_visible"
        }
      },
      "time": "2026-06-03T10:30:00.123+08:00"
    }
  ],
  "answerList": [],
  "beginTime": "2026-06-03T10:30:00.123+08:00",
  "endTime": "2026-06-03T10:31:20.456+08:00",
  "imgList": []
}
```

说明：

- `pageNumber`、`pageDesc`、`operationList`、`answerList`、`beginTime`、`endTime`、`imgList` 保持现有总体结构。
- `operationList[*].eventType` 使用本文档标准事件名，例如 `START_PAGE`、`TEXT_CHANGE`、`CHAT_SCROLL`。
- `operationList[*].time` 必须改为毫秒级 ISO 8601 时间。
- `operationList[*].value` 承载标准轨迹字段；如果现有传输层只允许字符串，则前端必须 `JSON.stringify(value)`，后端按结构化 JSON 解析，不能把所有信息塞成不可解析的自然语言。
- `field_registry_version` / `field_registry_hash` 与 `content_registry_version` / `content_registry_hash` 逻辑上属于页面级上下文；在当前 Mark JSON envelope 下，由 `START_PAGE.value.metadata` 承载，后端解析时注入当前页所有事件上下文，不要求每条事件重复上报。
- `page_index` 固定为 1-13。
- `page_id` 使用新标准稳定 ID。
- `legacy_page_id` 仅用于记录现有前端页面编号口径，不能作为新标准打标算法的主键。
- `field_registry_version` 表示本次事件使用的字段注册表版本；字段语义由静态 Field Registry 解释，不在事件流中上报。
- `content_registry_version` 表示本次事件使用的可阅读内容注册表版本；内容语义由静态 Content Registry 解释，不在事件流中重复上报材料正文或语义全文。
- `initial_visible_content_ids` 记录页面加载完成后初始视口内默认可见的主说明、题干、约束提示等内容块，仅表示学生有阅读机会，不表示已经阅读或理解；这些默认可见内容不需要额外生成 `CONTENT_EXPOSE`。
- `main_instruction_visible` 与 `viewport_state` 用于描述页面初始可见状态，帮助后端解释页面加载后的长停顿；不作为 L3 标签或 HMM 观测值。
- `field_registry_hash` 可选，用于前后端校验 registry 文件是否一致。
- `answerList` 是评分 / 答案判定的数据源，不替代过程事件；过程事件也不得替代 `answerList`。

### 3.2 `operationList` 中的标准轨迹事件

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `code` | number | 是 | 沿用现有 `operationList` 序号，页面内严格递增；后端映射为 `event_seq` |
| `targetElement` | string | 是 | 沿用现有目标元素字段，建议稳定英文控件 ID，并保留 `P{pageNumber}_` 前缀兼容 |
| `eventType` | enum | 是 | 标准 L2 事件类型，见第 4 节 |
| `time` | string | 是 | ISO 8601 时间，含时区，必须精确到毫秒；后端映射为 `TraceEvent.clientTime`，不要求在 `value` 中重复上报 `client_time` |
| `value.trace_id` | string | 是 | 单事件唯一 ID；推荐前端生成 UUID，若特殊环境无法生成，必须在联调时确认后端可由 `session_id + pageNumber + code` 稳定派生 |
| `value.page_id` | string | 是 | 标准页 ID |
| `value.page_type` | enum | 是 | 操作大类 + 拓扑子类，例如 `B2_TEXT_MULTI_PARALLEL` |
| `value.target_id` | string | 建议 | 去掉页面前缀后的控件稳定 ID |
| `value.target_type` | enum | 是 | `page` / `button` / `text` / `checkbox` / `radio` / `modal` / `experiment` / `chart` / `table` / `content` |
| `value.field_id` | string | 条件 | 文本输入、动态表格、题目答案等需要字段语义的事件必须携带；其语义由 Field Registry 查询 |
| `value.content_id` | string | 条件 | 聊天气泡、文字块、图表说明等内容查看事件必须携带；其语义由 Content Registry 查询 |
| `value.question_id` | string | 条件 | 选择题事件必须携带 |
| `value.option_id` | string | 条件 | 选择题、多选题事件必须携带 |
| `value.row_id` | string | 条件 | 第 12 页动态方案表事件必须携带 |
| `value.value_before` | any | 条件 | 交互前业务真实值 |
| `value.value_after` | any | 条件 | 交互后业务真实值 |
| `value.validation_status` | enum | 条件 | `success` / `blocked` / `auto` / `timeout` / `none` |
| `value.metadata` | object | 是 | 结构化附加信息，可为空对象 |

禁止把复杂信息压成不可解析的自然语言字符串；复杂信息必须在 `value.metadata` 中结构化存储。

时间精度要求：

- 所有事件级 `client_time` 必须精确到毫秒，推荐格式：`2026-06-03T10:30:10.123+08:00`。
- 当前 Mark JSON envelope 中的事件时间字段就是 `operationList[*].time`；后端解析后映射为内部 `client_time` / `clientTime`。
- 页面级 `client_page_begin_time`、`client_page_end_time` 也必须精确到毫秒。
- 禁止只上报秒级时间，例如 `2026-06-03 10:30:10`；秒级时间无法可靠计算停顿、输入片段、快速切换和重复点击。
- `event_seq` 仍然必须保留；当多个事件毫秒时间相同或客户端时间异常时，后端使用 `event_seq` 兜底排序。
- 现有秒级 Mark JSON 不纳入新标准轨迹打标，不得用补零方式伪造毫秒；若仅用于现状分析，必须显式标记 `metadata.time_precision="second"`。

### 3.3 字段语义边界

| 字段 | 应该放什么 | 不应该放什么 |
|---|---|---|
| `value_before` | 操作前的业务值，如聚焦时文本、改选前选项 | `success` / `blocked` 这类状态 |
| `value_after` | 操作后的业务值，如失焦最终文本、选中选项 | 大段过程日志、系统自动结果 |
| `validation_status` | 提交校验状态 | 题目答案、实验参数 |
| `metadata` | 字符数、参数快照、题号、行 ID、停留时长、触发原因等 | 能力判断结论、最终标签 |

### 3.4 L0 Raw Event 与 L2 Standard Event 边界

本文档第 4 节定义的是进入 `operationList` 的 L2 标准轨迹事件，不要求前端把浏览器原始事件逐条上报。

- L0 Raw Event：前端内部可采集的原始事件，如 `pointer_enter`、`pointer_leave`、`wheel`、`touch_start`、`intersection_observe`。这些事件用于计算停留、曝光、滚动段，不直接进入 HMM，也不直接等同于学生阅读理解。
- L2 Standard Event：前端在页面批量上报时写入 `operationList` 的标准事件，如 `TEXT_CHANGE`、`CHAT_SCROLL`、`CONTENT_EXPOSE`、`CONTENT_ACTIVATE`、`PAGE_IDLE`、`EXECUTE_EXP`。
- L3 Behavior Label：后端根据 L2 事件序列生成的行为标签，如 `READ_CHAT`、`REVIEW_CHAT`、`GENERATE_THINK`、`REVISE`。
- L4 Metadata：由事件聚合得到的状态与特征，如阅读覆盖率、关键气泡停留时长、字段完成度；默认不作为 HMM 主观测标签。

前端可以为了“尽可能完整地记录合理轨迹”增强 L0 采集，但必须在页面批量上报前完成去噪与聚合：快速划过、低可见比例、极短停留、连续滚动的逐像素变化都不应逐条进入 `operationList`。页面加载后或提交被拦截后的长时间无操作可以聚合为 `PAGE_IDLE`，但它只能表示阅读、理解任务、思考、犹豫或离开设备的候选状态，不能直接等同于阅读理解。

### 3.5 Field Registry（静态字段注册表）

`field_id` 的语义不由前端在事件流中动态上报，而由前后端共享的静态 Field Registry 解释。

原则：

- Field Registry 是版本化 schema 资源，例如 `field_registry_science_inquiry_v2.1.json` 或由同一份配置代码生成；不同子模块通过 `module_id` 和内容配置区分。
- 前端按 `page_id` 加载 registry 中声明的字段，事件只引用 `field_id`。
- 后端按 `field_registry_version + page_id + field_id` 查询 `field_role`、`required`、`method_slot`、`evaluation_side`、`content_key` 等语义。
- 字段注册表不进入 `operationList`、单条事件或 `START_PAGE.value.metadata` 的字段清单，避免每次会话重复上报，也避免把字段语义变成前端运行时责任。
- 若前端版本、后端版本和 registry 版本不匹配，应阻断上线或记录强告警；后端不得信任前端动态传来的字段语义覆盖 registry。
- 本标准负责定义会被打标算法消费的 `field_id`、`field_role`、`required`、`group_id`、`method_slot`、`evaluation_side`、`param_id`；前端实施 agent 负责把真实控件绑定到这些标准 ID。
- 页面展示文案、素材 ID、DOM ID、非打标分析字段可由前端按子模块实际情况定义；一旦字段会被后端行为打标消费，就必须进入 Field Registry，并接受本标准约束。

标准字段 ID 表：

| 页面 | 拓扑子类 | 标准 `field_id` | 关键语义 |
|---|---|---|---|
| Page02 | `B1_TEXT_SINGLE` | `input_question_1` | 科学问题文本输入，`field_role=text_input`，必填 |
| Page05 | `B2_TEXT_MULTI_PARALLEL` | `input_idea_1`、`input_idea_2`、`input_idea_3` | 3 个并列方案 / 想法文本输入，`field_role=text_input`，均必填 |
| Page06 | `B3_TEXT_MATRIX_EVALUATION` | `method_1_advantage`、`method_1_disadvantage`、`method_2_advantage`、`method_2_disadvantage`、`method_3_advantage`、`method_3_disadvantage` | 第 1/2/3 种方法 x 优点 / 缺点评价矩阵，`field_role=method_evaluation`，均必填；具体方法名称由模块 Field Registry 配置 |
| Page12 | `E1_CHART_PLAN_DECISION` | `chart_evidence_1`、`plan_table`、`reason_text` | 证据图表、方案表、全局理由文本；动态方案行用 `row_id` 追踪，行内参数用 `param_id=plan_param_1/plan_param_2/...`，不把具体主题参数名或动态行号硬编码进静态 `field_id` |

Field Registry 示例：

```json
{

  "task_family": "science_inquiry_13page",
  "module_id": "banana_browning",
  "pages": {
    "page_06_plan_evaluation": {
      "page_type": "B3_TEXT_MATRIX_EVALUATION",
      "required_fields": [
        "method_1_advantage",
        "method_1_disadvantage",
        "method_2_advantage",
        "method_2_disadvantage",
        "method_3_advantage",
        "method_3_disadvantage"
      ],
      "fields": [
        {
          "field_id": "method_1_advantage",
          "field_role": "method_evaluation",
          "method_slot": 1,
          "content_key": "banana_image_method",
          "display_name": "图像法",
          "evaluation_side": "advantage",
          "required": true,
          "order": 1,
          "group_id": "method_evaluation_matrix"
        }
      ]
    }
  }

```

实现建议：

- 前端构建时从 registry 生成 TypeScript 常量，避免手写 `field_id`。
- 后端构建或启动时加载同一份 registry，生成 `FieldRegistry` 查询服务。
- 现有 Mark JSON 仅用于差距分析，不为历史数据建立强制迁移映射；前端重构后的新事件必须直接引用 registry 中的标准 `field_id`。

### 3.6 Content Registry（静态内容注册表）

聊天气泡、说明文字、资料文本块、图表说明等“可阅读内容”的语义不应写在每条轨迹事件中，而应进入 Content Registry。

Content Registry 至少包含：

| 字段 | 说明 |
|---|---|
| `content_id` | 内容块稳定 ID，如 `chat_bubble_02_04` |
| `content_type` | `chat_bubble` / `instruction_text` / `material_text` / `chart_note` |
| `sequence_index` | 页面内呈现顺序 |
| `speaker_role` | 聊天气泡使用，如 `mother` / `student` / `system` |
| `is_key_evidence` | 是否为关键证据内容；不作为前端是否采集的筛选条件 |
| `evidence_role` | 关键证据角色，如 `problem_trigger`、`exclude_factor`、`task_instruction` |

原则：

- 所有聊天气泡都应注册 `content_id`，不能只注册关键气泡。
- 前端事件只引用 `content_id`，不重复上报完整文本。
- 关键气泡只影响后端解释权重和阅读特征，不影响前端是否采集。
- 内容曝光、鼠标停留、触摸停留只能作为阅读注意代理证据，不能直接证明阅读理解。

## 4. 标准事件字典

### 4.1 全局流程事件

| 事件 | 触发时机 | 关键字段 / Metadata |
|---|---|---|
| `START_PAGE` | 页面渲染完成，学生可以交互 | `page_role`、`from_page_id`、`flow_context`、`field_registry_version`、`field_registry_hash`、`content_registry_version`、`content_registry_hash`、页面初始状态；不记录字段清单或内容正文 |
| `PAGE_HIDDEN` | 页面进入后台、切 tab 或窗口失焦 | `hidden_reason` |
| `PAGE_VISIBLE` | 页面重新可见 | `hidden_duration_ms` |
| `PAGE_IDLE` | 页面可见、窗口聚焦、无有效交互持续达到阈值后，由前端聚合生成 | `idle_duration_ms`、`idle_phase`、`idle_start_event`、`idle_end_event`、`page_visible`、`window_focused`、`initial_visible_content_ids`、`next_event_type`、`cursor_region` |
| `SUBMIT_ATTEMPT` | 点击下一页 / 提交 / 自动提交 | `submit_attempt_id`、`validation_status`、`missing_fields`、`error_msg`、`submit_trigger` |
| `TASK_FINISH` | 第 13 页完成香蕉子模块 | `finish_trigger` |

`PAGE_IDLE` 实现规则：

- 不通过高频轮询生成，不逐秒上报，不监听或记录高频 `mousemove`。
- 前端只需维护上一个有效事件的时间、类型、页面可见状态和窗口聚焦状态；当下一个有效事件发生时，回算两者之间的无操作间隔。
- 仅当页面可见、窗口聚焦且无操作时长超过阈值，才生成一条聚合 `PAGE_IDLE`；页面结束时随当前页 `operationList` 批量上报。
- 建议初始阈值：页面加载后到首次有效操作 `>= 5000ms`；提交被拦截后到补救操作 `>= 3000ms`；普通有效操作之间 `>= 8000ms`。
- 页面隐藏或窗口失焦期间不生成有效 `PAGE_IDLE`；超长停顿可标记 `idle_quality=long_absence_candidate` 或写入 capped 时长，避免误判为深度思考。
- `PAGE_IDLE` 默认不进入 HMM 主观测序列；后端可将其作为页面定向、读说明、思考、犹豫、补救前停顿等候选证据。

### 4.2 材料查看与阅读事件

| 事件 | 触发时机 | 关键字段 / Metadata |
|---|---|---|
| `CONTENT_EXPOSE` | 非默认可见内容块进入可视区且达到有效曝光阈值 | `content_id`、`content_type`、`visible_ratio`、`visible_ms`、`phase` |
| `CONTENT_ACTIVATE` | 鼠标停留、触摸或点击某个内容块且达到有效激活阈值 | `content_id`、`content_type`、`activation_type`、`dwell_ms`、`phase` |
| `CONTENT_VIEW` | 学生主动查看、滚动、点击或展开页面文字材料的聚合事件 | `content_id`、`content_type`、`view_action`、`visible_content_ids`、`phase` |
| `CHAT_SCROLL` | 聊天材料区一次连续滚动段 | `scroll_delta`、`scroll_direction`、`visible_content_ids_before`、`visible_content_ids_after`、`phase` |
| `CHAT_VIEWPORT_ENTER` / `CHAT_VIEWPORT_LEAVE` | 鼠标进入 / 离开聊天手机框，可选增强采集 | `viewport_id`、`inside_ms`、`phase` |
| `OPEN_MODAL` | 打开资料卡、说明卡、详情卡 | `card_id`、`card_type` |
| `CLOSE_MODAL` | 关闭资料卡 | `card_id`、`dwell_ms` |
| `CHART_HOVER` | 第 12 页有效悬停图表节点 | `chart_id`、`point_id`、`hover_ms`、`data_snapshot` |

阅读类事件阈值建议：

- `CONTENT_EXPOSE`：可见比例 `>= 0.5` 且持续 `>= 500ms`；系统自动动画造成的曝光只能标记为 `expose_source=auto_reveal`，不直接作为主动阅读证据。
- `CONTENT_ACTIVATE`：鼠标停留 `>= 600ms` 或触摸停留 `>= 300ms`；短暂划过丢弃，`<= 100ms` 的 hover 断开可合并。
- `CHAT_SCROLL`：连续滚动合并为一个滚动段，滚动停止 `>= 300ms` 后落盘；不得逐像素上报。
- 这些事件只能作为阅读注意代理证据，不能直接等同于阅读理解。
- 页面加载后默认完整或基本完整可见的主说明文字、题干、约束提示，不单独生成 `CONTENT_EXPOSE`；这类内容写入 `START_PAGE.value.metadata.initial_visible_content_ids`。`CONTENT_EXPOSE` 主要用于需要滚动、展开、弹窗、tooltip 才能看到，或可能被遮挡 / 被小屏布局影响的内容。

### 4.3 文本输入事件

| 事件 | 触发时机 | 关键字段 / Metadata |
|---|---|---|
| `TEXT_FOCUS` | 输入框获得焦点 | `field_id`、`char_count_before` |
| `TEXT_CHANGE` | 防抖 / 节流后的文本变化 | `field_id`、`char_delta`、`char_count_after`、`trigger_reason` |
| `TEXT_BLUR` | 输入框失焦或提交前强制落盘 | `field_id`、`value_before`、`value_after`、`char_count_after` |

文本输入统一规则：

- 前端 **不得逐字实时上报** 每一次按键。
- `TEXT_CHANGE.value_after` 默认留空，不采集过程明文；过程特征写入 `metadata`。
- 最终完整文本只在 `TEXT_BLUR.value_after` 中上报。
- 中文输入法 composition 过程不作为独立事件；只在稳定文本变化后进入 `TEXT_CHANGE`。
- 推荐初始防抖：停止输入 2000ms 后上报一次。
- 推荐初始节流：累计字符变化 `abs(char_delta) >= 10` 时上报一次。

### 4.4 选择与判断事件

| 事件 | 触发时机 | 关键字段 / Metadata |
|---|---|---|
| `CHECKBOX_TOGGLE` | 第 3 页多选项勾选 / 取消 | `option_id`、`checked`、`selected_option_ids` |
| `SELECT_ANSWER` | 第 9-11 页单选题选择 | `question_id`、`question_group_id`、`question_index`、`option_id`、`option_text` |

### 4.5 模拟实验事件

| 事件 | 触发时机 | 关键字段 / Metadata |
|---|---|---|
| `SET_EXP_PARAM` | 修改实验参数，如天数、品种、温度 | `param_name`、`value_before`、`value_after`、`param_snapshot` |
| `EXECUTE_EXP` | 点击开始 / 执行实验 | `exp_run_id`、`param_snapshot`、`result_snapshot`、`click_debounce_applied` |
| `RESET_EXP` | 点击重置实验 | `param_snapshot_before_reset` |

注意：

- 系统自动渲染出的实验结果或指标数值不是学生主动操作，不作为独立事件。若后续分析需要使用实验结果，只能作为 `EXECUTE_EXP.metadata.result_snapshot`。
- 实验执行按钮初始防双击窗口为 `1500ms`。窗口内重复点击不生成新的有效实验执行，只在必要时记录 `click_debounce_applied=true`，阈值可在后续真实数据校准后版本化调整。

### 4.6 第 12 页方案表事件

| 事件 | 触发时机 | 关键字段 / Metadata |
|---|---|---|
| `ADD_ROW` | 新增方案行 | `row_id`、`row_index` |
| `DELETE_ROW` | 删除方案行 | `row_id`、`row_snapshot_before_delete`、`was_best_plan` |
| `SET_PLAN_PARAM` | 修改方案行参数 | `row_id`、`param_name`、`value_before`、`value_after`、`row_snapshot` |
| `SELECT_BEST` | 星标 / 标记最佳方案 | `row_id`、`previous_best_row_id`、`current_best_row_id` |

## 5. 合理采集与噪音抛弃标准

### 5.1 必须采集

满足任一条件即应采集：

- 会改变学生答案、方案、实验参数或选择状态。
- 能形成后端打标证据，如输入停顿、文本修正、资料回看、实验执行、作答后核对。
- 能修正时间解释，如页面隐藏 / 可见。
- 能解释提交失败，如空字段、未选择选项、倒计时未完成。

### 5.2 不应采集

| 噪音类型 | 处理 |
|---|---|
| 鼠标移动、无业务悬停 | 不采集 |
| 页面空白区域点击 | 不采集 |
| 系统自动动画、倒计时每秒变化 | 不采集；只采集 `TIMER_COMPLETE` 或提交拦截状态 |
| 输入法 composition 中间态 | 不采集 |
| 逐字输入事件 | 不逐字上报，合并为 `TEXT_CHANGE` |
| 重复快速点击开始实验 | 前端可上报，但必须带 `exp_run_id` 和参数快照，后端按规则去重 |
| 系统自动计算的实验结果展示 | 不作为学生行为事件；放入 `EXECUTE_EXP.metadata.result_snapshot` |
| 能力结论 | 前端禁止上报，由后端打标 / 分析生成 |

### 5.3 最小有效阈值

| 项 | 建议初始值 | 用途 |
|---|---:|---|
| `textDebounceMs` | 2000ms | 合并连续输入 |
| `textThrottleCharDelta` | 10 字符 | 长文本输入过程上报 |
| `modalMisclickMaxMs` | 300ms | 资料卡误开过滤 |
| `minMeaningfulDwellMs` | 800ms | 有效查看资料卡 / 信息 |
| `chartHoverMinMs` | 500ms | 第 12 页有效图表查看 |
| `focusDebounceMs` | 300ms | 同输入框焦点抖动过滤 |
| `expRunDebounceMs` | 1000ms | 同参数重复执行实验去重 |

这些阈值是 v1.0 初始工程值，不是理论常量。上线后应根据真实学生数据分布校准。

## 6. 操作行为大类与拓扑子类采集标准

### 6.1 A1 子类：流程 / 过渡 / 结束页

适用页面：

| 页面 | 标准页 ID | 页面角色 |
|---|---|---|
| 第 1 页 | `page_01_background` | 背景 / 任务进入 |
| 第 4 页 | `page_04_transition_to_design` | 阶段过渡 |
| 第 7 页 | `page_07_transition_to_experiment` | 实验阶段过渡 |
| 第 13 页 | `page_13_finish` | 香蕉子模块完成；不表示批次级监测结束 |

必采事件：

| 事件 | 必填 | 说明 |
|---|---:|---|
| `START_PAGE` | 是 | 记录页面到达和初始状态 |
| `PAGE_IDLE` | 条件 | 页面加载后或完成页停留达到阈值时上报 |
| `PAGE_HIDDEN` / `PAGE_VISIBLE` | 条件 | 切 tab 或窗口失焦时上报 |
| `SUBMIT_ATTEMPT` | 第 1、4、7 页是 | 点击下一页 |
| `TASK_FINISH` | 第 13 页是 | 香蕉子模块完成，不表示批次级监测结束 |

为什么这样采集：

- 这些页面主要是流程锚点，不承担核心作答任务。
- 采集到达、条件性长停顿和提交即可支持后端判断阶段完成与页面定向过程。
- `PAGE_HIDDEN` / `PAGE_VISIBLE` 可避免把离开页面的时间误判为思考时间。

禁止采集：

- 空白点击。
- 系统动画。
- 能力判断类字段。

### 6.2 B 类：文本生成页（P2 / P5 / P6 共用大类）

适用页面：第 2、5、6 页。

共同操作链路：

```text
查看页面材料 / 任务说明
→ 构思或停顿
→ 聚焦输入框
→ 输入文本
→ 修改文本
→ 失焦或提交前落盘
→ 提交尝试
→ 可能回看材料或切换字段检查
```

共同必采事件：

| 事件 | 说明 |
|---|---|
| `START_PAGE` | 页面启动，记录页面初始状态、registry 版本和页面上下文；不记录字段清单 |
| `PAGE_IDLE` | 页面加载后到首次有效操作、提交被拦截后到补救操作等达到阈值的长停顿 |
| `CONTENT_VIEW` / `CHAT_SCROLL` | 学生主动查看、滚动、点击或回看材料；第 2 页聊天区可继续使用 `CHAT_SCROLL` |
| `TEXT_FOCUS` | 输入框获得焦点 |
| `TEXT_CHANGE` | 防抖 / 节流后的输入变化 |
| `TEXT_BLUR` | 失焦或提交前最终文本快照 |
| `SUBMIT_ATTEMPT` | 点击下一页，记录校验状态 |
| `PAGE_HIDDEN` / `PAGE_VISIBLE` | 条件上报，用于修正停留和停顿时长 |

共同文本规则：

- 三个页面共用第 4.3 节文本事件定义。
- 前端不逐字上报，不上报 `GENERATE_THINK`、`GENERATE_FLUID`、`REVISE` 等后端标签。
- `TEXT_CHANGE.value_after` 默认留空，字符数、字符变化量、触发原因写入 `metadata`。
- `TEXT_BLUR.value_before` 为聚焦时完整文本，`TEXT_BLUR.value_after` 为失焦时完整文本。
- 页面加载后到首次有效操作前、提交被拦截后到补救操作前的长时间无操作，可以由前端按第 4.1 节聚合为 `PAGE_IDLE`。输入框内部的细粒度停顿仍由后端根据 `TEXT_FOCUS`、`TEXT_CHANGE`、`TEXT_BLUR`、`PAGE_HIDDEN` / `PAGE_VISIBLE` 的时间差计算。

#### 6.2.1 B 类通用 Field Registry 规则

B 类页面的字段拓扑必须在第 3.5 节 Field Registry 中静态声明。前端事件只携带 `field_id`，不在 `START_PAGE.value.metadata`、页面 Payload 或单条事件中上报完整字段清单；`START_PAGE.value.metadata` 只允许承载 registry 版本、hash 和页面上下文。

字段注册项说明：

| 字段 | 说明 |
|---|---|
| `field_id` | 文本框稳定 ID，文本事件必须引用 |
| `field_role` | 字段角色，如 `text_input`、`method_evaluation`、`reason_text` |
| `required` | 是否必填 |
| `order` | 页面呈现顺序 |
| `group_id` | 字段组，如 `plan_ideas`、`method_evaluation_matrix` |
| `method_slot` | 第 6 页使用，表示页面中的第几种方法；具体方法名称由模块 Field Registry 配置 |
| `evaluation_side` | 第 6 页使用，表示 `advantage` / `disadvantage` |

为什么使用 Field Registry：

- 保持文本事件简洁，避免每条 `TEXT_CHANGE` 都携带大量 nullable 字段。
- 保留第 2、5、6 页不同控件拓扑，避免把差异隐藏到后端硬编码。
- 支持前端复用同一个 `TextEventCollector`，只替换静态注册表中的字段清单和必填集合。
- 避免每次会话重复上报字段清单，降低带宽和版本不一致风险。

#### 6.2.2 B1 子类：单输入框文本生成页

适用页面：第 2 页，`page_02_question_generation`。

## 7. 现有标准差距与重构要求

### 7.1 当前前端实现事实

当前 `香蕉变黑轨迹数据现有标准.md` 使用 Mark JSON 页面级结构。该结构只作为现状证据和差距分析依据，不作为新标准历史迁移对象：

```json
{
  "pageNumber": "1.03",
  "pageDesc": "香蕉的奥秘",
  "operationList": [],
  "answerList": [],
  "beginTime": "2026-06-03 10:30:00",
  "endTime": "2026-06-03 10:30:20",
  "imgList": []
}
```

当前单条操作结构：

```json
{
  "code": 1,
  "targetElement": "P1.03_科学问题输入",
  "eventType": "input_change",
  "value": "{\"prev\":\"\",\"next\":\"为什么\"}",
  "time": "2026-06-03 10:30:10"
}
```

当前实现要点：

- `operationList` 只含 `code`、`targetElement`、`eventType`、`value`、`time`。
- `beginTime`、`endTime`、`time` 当前按 `YYYY-MM-DD HH:mm:ss` 秒级记录，精度不足以支持停顿、快速切换、重复点击等过程分析。
- `targetElement` 使用 `P{pageNumber}_{元素名称}` 格式。
- 文本输入当前是 `input_focus → input_change(每次按键) → input_blur`。
- `input_change.value` 当前可能保存 `{ prev, next }` 过程明文。
- 香蕉变黑实例当前主要手动调用 `logOperation()`，未完全使用共享 `TrackedInput` / `TrackedTextarea`；其他同构子模块重构时也应避免继续扩散这种页面内手写日志模式。
- 页面配置和题目编码已有静态配置基础，如 `mapping.ts`、`QUESTION_CODE_MAP`、`QUESTION_TEXT_MAP`，Field Registry 应在这个方向上扩展，而不是作为运行时事件上报。

### 7.2 新标准重构要求

旧数据不做历史迁移。前端完成标准建立后，应对香蕉变黑及其他同构子模块逐步重构，让新产生的数据直接符合本文档 schema。

重构时必须满足：

| 当前事实 | 新标准要求 | 原因 |
|---|---|---|
| 页面级 envelope 已有 `pageNumber`、`pageDesc`、`operationList`、`answerList` 等字段 | 保持不变，只重构 `operationList[*]` 的事件名、时间精度和 `value` 结构 | 避免打破当前前端整体上报和评分链路 |
| `targetElement` 是中文控件描述 | 保留字段名，但改为稳定控件 ID；标准 `field_id`、`content_id`、`question_id`、`option_id`、`row_id` 放入 `value` | 打标算法需要稳定机器可读字段，不能依赖文案解析 |
| `eventType` 是旧事件名 | 保留字段名，但取值改为本文档标准 L2 事件名 | 前后端事件字典必须一致，同时不破坏现有 operation 字段名 |
| `input_change` 逐字上报 | 改为防抖 / 节流后的 `TEXT_CHANGE` | 逐字事件噪音大，且会暴露不必要的过程明文 |
| `time` 只有秒级 | 保留字段名，但取值必须为毫秒级 ISO 8601 带时区时间 | 停顿、快速切换、双击去重和页面隐藏时长都需要毫秒精度 |
| 字段语义隐含在控件名里 | 字段语义进入静态 Field Registry | 防止前端运行时动态语义覆盖后端契约 |

### 7.3 `answerList` 与轨迹打标边界

`answerList` 必须保留，但它不属于轨迹打标输入。

- `answerList` 是评分 / 答案判定的数据源，所有评分所需答案均从 `answerList` 获取。
- `operationList` 中按新标准结构化后的轨迹事件是行为轨迹打标的数据源，后端只用它们生成过程标签。
- `TEXT_BLUR.value_after` 可作为“学生在某时刻输入了什么”的过程证据，但不能替代 `answerList` 成为评分答案源。
- 打标算法不得为了判断答案正确性而读取或分析 `answerList`；如发现打标实现读取 `answerList`，应输出数据质量 / 实现告警。

### 7.4 字段 ID 重构验收

前端实施 agent 可以根据真实代码调整控件绑定方式，但不得擅自改变本标准定义的 `field_id`。验收时至少检查：

- Page02 文本输入事件必须使用 `field_id=input_question_1`。
- Page05 三个文本输入事件必须使用 `input_idea_1`、`input_idea_2`、`input_idea_3`。
- Page06 六个评价输入事件必须使用 `method_1_advantage`、`method_1_disadvantage`、`method_2_advantage`、`method_2_disadvantage`、`method_3_advantage`、`method_3_disadvantage`。
- Page12 图表、方案表、理由文本分别使用 `chart_evidence_1`、`plan_table`、`reason_text`；动态方案行只通过 `row_id` 关联，方案参数通过通用 `param_id=plan_param_1/plan_param_2/...` 关联。
- `SUBMIT_ATTEMPT.validation_status=blocked` 时必须提供 `missing_fields`；需要给学生或后端解释失败原因时提供 `error_msg`；每次提交尝试必须有 `submit_attempt_id`。

## 8. 合规检查清单

前端上线前必须检查：

- 13 个页面均有稳定 `page_id`、`page_index`、`page_type`。
- 前后端使用同一份 Field Registry，并校验 `field_registry_version` / `field_registry_hash`。
- 所有事件有 `trace_id`、`event_seq`、毫秒级 `client_time`。
- `START_PAGE.metadata.initial_visible_content_ids` 能记录页面初始可见的主说明、题干、约束提示；默认可见内容不重复生成 `CONTENT_EXPOSE`。
- 条件性 `PAGE_IDLE` 不通过高频轮询、逐秒上报或记录 `mousemove` 生成；只在有效事件发生时回算无操作间隔，并校验 `page_visible=true`、`window_focused=true`。
- 文本、题目、动态表格等关键事件能上报稳定 `field_id` / `question_id` / `option_id` / `row_id`。
- 文本输入不逐字上报，`TEXT_CHANGE.value_after` 默认留空。
- `TEXT_BLUR` 能上报最终完整文本。
- `SUBMIT_ATTEMPT.validation_status` 不写入 `value_after`。
- 第 8 页不出现 `SELECT_ANSWER`。
- 第 9-11 页每页只上报当前页当前题的 `SELECT_ANSWER`。
- 实验结果只进入 `EXECUTE_EXP.metadata.result_snapshot`。
- 动态表格行使用稳定 `row_id`。
- 前端不上报任何能力判断、策略判断、标签结论。
- 前端不得上报 L3 行为标签或 L4 过程特征；本阶段只交付 L0/L1 到 L2 的标准化上报。
- 当前香蕉实现使用 `engineering_contracts/event_schema.v2.2.json` 对 `operationList[*]` 做本地校验，使用 `engineering_contracts/field_registry.v2.2.json` 和 `engineering_contracts/content_registry.banana.v2.2.json` 校验标准 ID；v2.1 合同仅用于兼容回放 v2.1 系列样本。
